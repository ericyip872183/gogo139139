## 一、最高优先级规则 ⚠️

### 🔴 规则 0：分级授权机制

**核心原则**：一次确认，全程授权，异常暂停，结果汇报

**执行流程**：
1. **提供方案**：收到任务后，先提供完整方案
2. **获取授权**：等待用户确认，确认后获得"执行授权"
3. **自主执行**：在方案范围内自主决策，无需重复确认
4. **异常暂停**：遇到方案外问题，立即暂停并汇报
5. **结果汇报**：完成后汇报执行结果和关键变更

**授权范围（确认后无需再确认）**：
- ✅ 创建/修改/删除文件
- ✅ 编写/修改/重构代码
- ✅ 自测验证
- ✅ 更新文档（PROGRESS.md、API.md 等）
- ✅ 清理临时文件

**需要额外确认的操作**：
- ⚠️ 数据库迁移（可能影响生产数据）
- ⚠️ Git 提交（用户可能想先查看变更）
- ⚠️ 部署操作（影响线上环境）
- ⚠️ 删除大量文件（不可逆操作）

**示例**：
```
用户：实现用户管理 CRUD
AI：提供方案 → 用户确认 → 获得执行授权
    ↓
    自主执行：
    ✓ 创建后端接口文件
    ✓ 编写 CRUD 代码
    ✓ 创建前端页面
    ✓ 自测通过
    ✓ 更新 PROGRESS.md
    ↓
    完成：汇报结果 + 展示关键变更
```

## 二、协作习惯

### 2.1 文档语言

**偏好**：文档内容使用中文，方便阅读

> 文件名可以是英文，但文件内容尽量是中文

### 2.2 代码注释

**要求**：每个新建文件顶部必须有简介注释

```typescript
/**
 * 文件：xxx.service.ts
 * 说明：XXX 管理服务（数据库操作）
 * 权限：TEACHER+
 * 对应接口：GET/POST/PATCH/DELETE /api/xxx
 */
```

### 2.3 提交习惯

**偏好**：提交前需要先给用户确认

> 不要擅自提交代码，先给用户看 diff

### 2.4 调试习惯

**要求**：功能完成后必须自测，不要让用户当测试员

> 写完代码后自己先跑一次，把日志记录下来

## 三、开发规范偏好

### 3.1 TypeScript

- ✅ 严格模式，禁止使用 `any`
- ✅ 每个函数/接口必须有类型定义
- ✅ 使用接口（interface）而非类型别名（type）定义对象

### 3.2 多租户隔离

- ✅ 所有数据库查询必须带 `tenantId`
- ✅ `tenantId` 从 JWT Token 中提取

### 3.3 API 响应

- ✅ 统一响应格式：`{ code, message, data, timestamp }`
- ✅ 分页响应：`{ total, list, page, pageSize }`

### 3.4 文件命名

- 后端模块：`xxx.module.ts`, `xxx.controller.ts`, `xxx.service.ts`
- 前端页面：`XxxView.vue`
- 调试脚本：`test-xxx.js`

**适用范围**：所有开发任务
