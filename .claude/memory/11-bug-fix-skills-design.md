### 2.2 技能记录格式

```markdown
## 🔥 TOP 优先级技能（出现 2 次以上）

### [TOP-1] API 响应格式不匹配
**触发场景**：API 返回数据结构变化，代码解析失败
**典型特征**：
- 错误信息：`Cannot read property 'xxx' of undefined` / `格式异常`
- 控制台有数据返回，但代码抛错
- 第三方 API 升级/迁移后首次使用

**解决模式**：
1. 先打印完整响应：`console.log(JSON.stringify(data, null, 2))`
2. 检查响应结构是否与文档一致
3. 更新解析逻辑，优先处理数组和可选链

**案例**：
- 2026-03-29：火山引擎 Responses API 响应解析错误
  - 问题：`output.text` 不存在，实际在 `output[].content[0].text`
  - 解决：添加数组检测 `Array.isArray(data.output)`

**优先级**：⭐⭐⭐⭐⭐（最高）
**命中率**：5 次 / 3 个月
```

### 2.3 普通技能区

```markdown
## 📚 技能库（按日期倒序）

### [SKILL-001] TypeScript 类型推断失败
**日期**：2026-03-29
**场景**：变量可能未赋值，TS 报错"used before being assigned"
**症状**：
```
error TS2454: Variable 'apiKey' is used before being assigned
```
**根因**：条件分支过多，TS 无法确定所有路径都赋值
**解决方案**：
```typescript
// ❌ 错误
let apiKey: string
if (condition1) apiKey = value1
if (condition2) apiKey = value2

// ✅ 正确
let apiKey: string | undefined
if (condition1) apiKey = value1
else if (condition2) apiKey = value2
if (!apiKey) throw new Error('未配置')
```
**适用范围**：所有 TypeScript 项目
**首次出现**：2026-03-29
**出现次数**：1 次
```

## 三、工作流程设计

### 3.1 修复前流程（强制执行）

```
用户报告 Bug
    ↓
读取 11-bug-fix-skills.md
    ↓
匹配 TOP 优先级技能（关键词/症状）
    ├─ 匹配成功 → 直接应用解决方案
    └─ 无匹配 → 读取 07-restart-solutions.md
           ↓
       常规调试流程
```

**匹配规则**：
1. 先匹配 **TOP 优先级**（命中率 > 2 次）
2. 再匹配 **关键词**（错误信息、技术栈、症状）
3. 最后匹配 **场景**（启动/部署/运行时/API）

### 3.2 修复后流程（自动执行）

```
Bug 解决
    ↓
提取修复模式（精简为 3 句话）
    ├─ 检查是否已有类似技能
    │   ├─ 有 → 提升优先级，合并到 TOP 区
    │   └─ 无 → 新增到普通技能库
    ↓
更新文档元数据（版本号、日期、技能数）
    ↓
生成「精简方案」提交用户确认
    ├─ 新增了什么技能
    ├─ 哪些技能提升了优先级
    └─ 文档进化方向
```

## 四、精简与进化机制

### 4.1 每日总结规则

**触发时机**：每次会话结束时

**精简策略**：
```markdown
1. 合并同类项
   - 同一 Bug 出现 2 次 → 标记 ⭐
   - 同一 Bug 出现 3 次 → 升级 TOP 优先级
   - 相似技能合并为"解决模式"

2. 删除冗余
   - 删除重复描述，保留核心步骤
   - 删除过时技术栈（如 Vue 2 → Vue 3）
   - 删除已失效的解决方案

3. 优化表达
   - 长段落 → 表格/流程图
   - 代码示例保留核心片段（<10 行）
   - 错误信息用反引号标记
```

### 4.2 版本演进记录

```markdown
## 版本历史

| 版本 | 日期 | 技能数 | TOP数 | 进化说明 |
|------|------|--------|-------|---------|
| v1.0 | 2026-03-28 | 8 | 0 | 初始建立，散乱记录 |
| v1.1 | 2026-03-29 | 12 | 1 | 发现 API 格式问题高频，首次设立 TOP |
| v1.2 | 2026-03-29 | 14 | 2 | 合并 Docker 相关问题，精简描述 |
| v1.3 | 2026-03-29 | 15 | 3 | 新增 TypeScript 类型问题，优化匹配规则 |

**进化方向**：
- v1.0 → v1.1：从被动记录到主动发现高频问题
- v1.1 → v1.2：从冗长描述到精简模板

```

## 六、实施步骤

### 6.1 立即执行（本次会话）

1. 创建 `.claude/memory/11-bug-fix-skills.md`
2. 迁移 `07-restart-solutions.md` 中的代码级 Bug 到新文件
3. 记录本次 Responses API 修复经验
4. 更新 `MEMORY.md` 索引

### 6.2 后续维护（每次会话）

1. **修复前**：读取 `11-bug-fix-skills.md` 匹配技能
2. **修复后**：提取模式 → 更新文档 → 提交方案
3. **会话结束**：检查是否需要精简（技能数 > 20 触发）

### 6.3 质量保证

**强制规则**：
- 每条技能必须包含：触发场景 + 典型特征 + 解决模式 + 案例
- 代码示例不超过 10 行
- 长描述转换为表格/列表
- TOP 优先级必须包含"命中率"统计

## 九、风险与对策

### 9.1 文档膨胀风险

**症状**：技能数 > 50，查找效率下降

**对策**：
- 自动合并相似技能
- 删除命中率 < 10% 的技能
- 按技术栈分类（前端/后端/部署）

### 9.2 过时技能风险

**症状**：解决方案在新版本失效

**对策**：
- 每条技能标注"适用版本"
- 用户反馈失效时立即标记"已过时"
- 定期清理（每季度）

## 十、总结

### 核心价值

1. **避免重复踩坑**：同样的 Bug 只需调试一次
2. **形成调试直觉**：TOP 技能直接应用，无需思考
3. **知识资产积累**：技能库可迁移到其他项目

### 实施关键

- **修复前必读**：强制流程，养成习惯
- **精简描述**：3 句话说清楚核心
- **用户确认**：保证质量，避免膨胀
- **每日进化**：持续优化，越用越准
