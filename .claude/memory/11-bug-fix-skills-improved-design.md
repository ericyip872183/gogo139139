# Bug 修复记忆系统设计方案（改进版 v2）

> **核心改进**：区分"通用经验"与"项目特定经验"，支持自动化迁移

---

## 一、问题分析

### 1.1 当前设计的问题

```
问题场景：
┌─────────────────────────────────────┐
│ 项目 A：Vue 3 + NestJS + MySQL      │
│ 技能库：15 个技能                    │
│  ├─ API 格式解析（通用）✅           │
│  ├─ TypeScript 类型（通用）✅        │
│  ├─ Docker 端口冲突（通用）✅        │
│  ├─ Prisma 连接问题（项目特定）❌    │
│  ├─ TinyMCE 配置（项目特定）❌       │
│  └─ Volcano Engine API（项目特定）❌ │
└─────────────────────────────────────┘
          ↓ 复制到新项目
┌─────────────────────────────────────┐
│ 项目 B：React + Express + MongoDB   │
│ 载入 15 个技能，但只有 8 个通用      │
│ ❌ 每次都要手动删除 7 个不相关技能   │
└─────────────────────────────────────┘
```

### 1.2 用户提出的解决方案

**核心思路**：
1. 分离"通用"和"项目特定"
2. 创建迁移提示词
3. 新项目首次执行自动清理
4. 清理完成后删除提示词

**优点**：
✅ 避免每次手动删除
✅ 新项目启动干净
✅ 保留高价值通用经验

## 二、改进方案对比

### 方案 A：单文件 + 标记区分（推荐）⭐

```markdown
# 11-bug-fix-skills.md

## 🔥 TOP 优先级技能（通用）

### [TOP-1] API 响应格式不匹配
**适用范围**：🌐 通用（所有项目）
**技术栈**：HTTP API / JSON
...

### [TOP-2] TypeScript 类型推断失败
**适用范围**：🌐 通用
**技术栈**：TypeScript
...

---

## 📚 项目特定技能

### [PROJ-001] Prisma 连接 MySQL 失败
**适用范围**：🎯 项目特定（NestJS + Prisma + MySQL）
**技术栈**：Prisma / MySQL / Docker
...

### [PROJ-002] Volcano Engine Responses API 调用
**适用范围**：🎯 项目特定（火山引擎）
**技术栈**：火山引擎 / Responses API
...
```

**优点**：
- ✅ 单文件，易于维护
- ✅ 标记清晰，一目了然
- ✅ 自动化脚本简单

**缺点**：
- ⚠️ 文件可能较长

---

### 方案 B：双文件分离

```
记忆系统结构：
├── 11-bug-fix-skills-common.md      # 通用经验（所有项目）
└── 11-bug-fix-skills-project.md     # 项目特定（当前项目）
```

**优点**：
- ✅ 迁移时只复制 common 文件
- ✅ 职责分离明确

**缺点**：
- ❌ 需要维护两个文件
- ❌ TOP 优先级跨文件难以排序
- ❌ 查找时需要读取两个文件

---

### 方案 C：智能识别（不推荐）

**思路**：不标记，用 AI 自动识别哪些是通用的

**缺点**：
- ❌ 准确率无法保证
- ❌ 每次迁移都要 AI 判断
- ❌ 不可控

---

### 🎯 最终推荐：方案 A（单文件 + 标记）

## 三、迁移机制设计

### 3.1 迁移提示词存储位置

**方案对比**：

| 位置 | 优点 | 缺点 |
|------|------|------|
| MEMORY.md 顶部 | 首次必读 | 可能被忽略 |
| 单独文件 `00-migration-prompt.md` | 醒目 | 额外文件 |
| 11-bug-fix-skills.md 顶部 | 紧密关联 | 每次载入都看到 |

**推荐方案**：`00-migration-prompt.md`（独立文件，执行后删除）

---

### 3.2 迁移提示词内容

```markdown
# 🚀 新项目首次执行提示词

> **触发条件**：复制 `.claude/memory/` 目录到新项目后首次启动

---

## 执行步骤

### 第一步：读取当前项目技术栈

请询问用户或检查项目文件，确定以下信息：

```
前端框架：React / Vue / Angular / 其他
后端框架：Express / NestJS / Django / 其他
数据库：MySQL / MongoDB / PostgreSQL / 其他
部署方式：Docker / K8s / 传统部署
特殊服务：AI 服务商 / 云存储 / 消息队列
```

### 第二步：清理项目特定技能

打开 `11-bug-fix-skills.md`，执行以下规则：

**保留规则**：
- 标记为 `🌐 通用` 的所有技能
- 技术栈匹配当前项目的 `🎯 项目特定` 技能

**删除规则**：
- 技术栈不匹配的 `🎯 项目特定` 技能
- 案例：删除 `Prisma` 相关技能（如果新项目用 MongoDB）

**示例**：
```markdown
❌ 删除：
### [PROJ-001] Prisma 连接 MySQL 失败
**适用范围**：🎯 项目特定（NestJS + Prisma + MySQL）

✅ 保留：
### [TOP-1] API 响应格式不匹配
**适用范围**：🌐 通用
```

### 第三步：更新文档元数据

更新 `11-bug-fix-skills.md` 头部：

```markdown
---
version: v1.0
lastUpdated: 2026-03-29
totalSkills: 8      ← 更新为清理后的数量
topPriority: 3
project: "新项目名称"
---
```

### 第四步：删除本提示词文件

执行完成后，删除 `00-migration-prompt.md`，避免每次载入重复执行。

### 4.2 标记规范

**通用技能标记**：
```markdown
**适用范围**：🌐 通用（所有项目）
**技术栈**：TypeScript / HTTP API / Docker
```

**项目特定技能标记**：
```markdown
**适用范围**：🎯 项目特定（NestJS + Prisma + MySQL）
**技术栈**：Prisma / MySQL / Docker
```

---

### 4.3 判断标准

**通用经验特征**：
- ✅ 与具体技术栈无关（如 API 格式、类型系统）
- ✅ 主流技术通用问题（如 Docker、TypeScript、Git）
- ✅ 调试方法论（如日志分析、错误定位）

**项目特定经验特征**：
- 🎯 依赖特定库/服务（如 Prisma、Volcano Engine）
- 🎯 依赖特定架构（如 Monorepo、微服务）
- 🎯 依赖特定云服务（如 阿里云 OSS、企业微信）

---

    │   ├─ 删除不匹配的项目特定技能
    │   ├─ 更新文档元数据
    │   └─ 删除 00-migration-prompt.md

```

## 六、优化细节

### 6.1 避免误删

**问题**：有些技能看起来是项目特定，但实际通用

**示例**：
```markdown
### [SKILL] Docker 端口冲突
**适用范围**：🎯 项目特定（Panlei 用 Docker）

❌ 错误判断：删除
✅ 正确判断：这是通用问题，保留
```

**解决方案**：
- 删除前询问用户确认
- 提供预览列表：`将删除以下技能：[PROJ-001] Prisma...`
- 用户可选择保留

---

### 6.2 迁移后学习

**问题**：新项目可能有新的通用技能

**示例**：
```markdown
# 项目 B 遇到 React Hooks 依赖问题
解决方案：useEffect 依赖数组

# 这是通用技能吗？
✅ 是，所有 React 项目都适用
→ 标记为 🌐 通用
```

**解决方案**：
- 新增技能时默认询问"是否通用？"
- 提供 `[通用] [项目特定]` 选项

---

### 6.3 多项目维护

**问题**：用户同时在维护多个项目

**示例**：
```
项目 A（Panlei）：Vue + NestJS
项目 B（新项目）：React + Express
```

**解决方案**：
- 每个项目独立的 `.claude/memory/`
- 定期同步通用技能到个人知识库
- 提供"导出通用技能"功能
