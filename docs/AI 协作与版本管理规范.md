# 若容平台 — AI 协作与版本管理规范

> **核心思想**：用 Git 管理代码版本，用文档管理上下文，用模板规范沟通。
> 确保任何时候（哪怕隔了几个月）回来，都能快速进入状态，不乱、不丢、不重复。

---

## 第一部分：为什么需要这套规范

### 1.1 我们经历过的问题

| 问题 | 后果 | 根本原因 |
|------|------|----------|
| 改了源代码没重新构建 | 服务器显示旧内容 | 没有构建验证步骤 |
| 上传文件不删除旧的 | 服务器堆积大量旧文件 | 没有清理机制 |
| 后端代码没同步 | API 返回 404 | 没有完整部署清单 |
| JWT_SECRET 不一致 | Token 验证失败 | 配置管理混乱 |
| nginx CORS 缺失 | 浏览器拦截请求 | 缺少配置检查 |
| 没有版本记录 | 不知道改了什么 | 没有 Git 提交历史 |

### 1.2 解决方案概览

```
┌─────────────────────────────────────────────────────────┐
│                    会话开始                             │
│  1. 读 CLAUDE.md → 了解项目架构                          │
│  2. 读 docs/PROGRESS.md → 了解功能进度                   │
│  3. 读 docs/AI 协作规范.md → 了解本次会话目标              │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                    开发流程                             │
│  1. 用沟通模板明确需求                                   │
│  2. 按模块逐个开发（不跨模块）                           │
│  3. 完成立即提交 Git + 更新进度文档                       │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                    会话结束                             │
│  1. 更新 docs/PROGRESS.md                                │
│  2. Git 提交 + 打标签                                      │
│  3. 写会话摘要到 docs/SESSIONS.md                        │
└─────────────────────────────────────────────────────────┘
```

## 第三部分：版本管理流程

### 3.1 日常开发流程（每次修改代码）

```bash
# 1. 开发前确认状态
cd c:\项目\panlei
git status
# 确保 "working tree clean"，否则先提交

# 2. 开发功能（按模块逐个开发）
# ... 编码 ...

# 3. 本地构建测试
pnpm run build:frontend

# 4. 提交代码
git add .
git commit -m "feat: [模块名] 完成 [功能描述]"
# 例如：feat: questions 完成题库批量导入功能

# 5. 更新进度文档
# 编辑 docs/PROGRESS.md，将对应功能从 ❌ 改为 ✅
```

### 3.2 部署流程（开发完成要上线）

```bash
# 1. 确认代码已提交
git status
# 应该显示 "working tree clean"

# 2. 查看当前版本
git log -1 --oneline

# 3. 打版本标签
git tag -a v1.x.x -m "版本说明"
# 例如：git tag -a v1.2.0 -m "新增体质辨识问卷功能"

# 4. 运行部署脚本
./deploy/sync-to-server.sh

# 5. 验证部署
curl https://pl.7025700.vip | grep title
# 应该显示：若容虚拟数据平台
```

### 3.3 避免旧文件堆积

**核心方法**：每次上传前先清空服务器旧文件

部署脚本中已包含：
```bash
# 清空服务器旧文件
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"

# 上传新文件
scp -r packages/frontend/dist/* root@120.77.144.5:/var/www/panlei/
```

**为什么这样做？**
- Vite 每次构建生成的 JS/CSS 文件名都不同（带哈希）
- 旧文件名如 `index-ABC.js`，新文件名如 `index-DEF.js`
- 不删除旧文件，服务器上会堆积数十个旧版本
- 清空 assets 目录确保只有当前版本

### 3.4 版本号命名规则

```
v<主版本>.<次版本>.<修订版本>
   │       │       │
   │       │       └─ Bug 修复（小问题，不影响功能）
   │       └─────  新功能（向后兼容）
   └─────────────  重大变更（不兼容旧版本）
```

| 版本号 | 何时使用 | 示例 |
|--------|---------|------|
| v1.0.0 | 首次上线 | 平台正式发布 |
| v1.0.1 | 修复 Bug | 修复登录页面样式问题 |
| v1.1.0 | 新增功能 | 新增题库管理模块 |
| v2.0.0 | 重构架构 | 数据库表结构大改 |

## 第四部分：大型项目开发规范

### 4.1 模块开发顺序

**永远一次只开发一个功能模块**，顺序：

```
1. 读 docs/PROGRESS.md → 确认哪些功能已完成，哪些待开发
         ↓
2. 选择下一个要开发的功能模块
         ↓
3. 读对应模块设计文档 (MODULES_CORE.md / MODULES_SPECIALTY.md)
         ↓
4. 读 DATABASE.md → 确认涉及哪些表
         ↓
5. 读 API.md → 确认接口规范
         ↓
6. 开始开发（先后端，后前端）
         ↓
7. 开发完成 → 自测 → 提交 Git → 更新进度文档
```

### 4.2 不跨模块开发

**错误做法**：
> "今天把题库管理和试卷管理一起做了"

**正确做法**：
> "今天只做题库管理的批量导入功能，做完提交，明天再做试卷管理"

**为什么？**
- 跨模块开发容易遗漏细节
- 出问题难以定位是哪个模块的问题
- Git 提交信息不清晰，无法回滚到中间状态
- AI 上下文容易混乱

### 4.3 文件组织规范

```
panlei/
├── docs/                      # 所有文档放这里
│   ├── PROGRESS.md           # 功能进度（每次更新）
│   ├── API.md                # 接口文档（新增接口时更新）
│   ├── DATABASE.md           # 数据库文档（改表结构时更新）
│   ├── AI 协作规范.md          # 本文档
│   └── SESSIONS.md           # 会话历史记录（每次会话后追加）
├── deploy/                    # 部署脚本
│   └── sync-to-server.sh     # 一键部署
└── packages/
    ├── frontend/src/views/   # 前端页面（按模块分目录）
    └── backend/src/modules/  # 后端模块（按功能分目录）
```

### 4.4 代码清晰度保障

**三条铁律**：

1. **不写注释**（除非逻辑极其复杂）
   - 好的代码自己会说话
   - 只有算法/业务规则特别复杂时才加注释

2. **不保留未使用的代码**
   - 删掉的代码不要注释保存，Git 会记住历史
   - 临时调试代码用完立即删除

3. **一个函数只做一件事**
   - 函数超过 50 行就要考虑拆分
   - 一个文件超过 500 行就要考虑拆分

### 5.1 暂停前（每次离开项目前）

```markdown
## 会话暂停记录

暂停时间：2026-03-21 22:00

本次会话完成：
- [x] 体质辨识问卷前端页面
- [x] 体质辨识问卷后端接口

下次从哪里继续：
- 从「体质辨识结果页面」继续
- 涉及文件：packages/frontend/src/views/constitution/ResultView.vue

待办事项：
- [ ] 结果雷达图组件
- [ ] 体质类型判定逻辑
- [ ] 结果页面 UI 优化
```

**操作**：
1. 将以上内容追加到 `docs/SESSIONS.md`

## 第七部分：与 AI 沟通的最佳实践

### 7.1 说什么（DO）

✅ **清晰描述目标**
> "我要开发体质辨识模块的结果页面，显示雷达图和体质类型"

✅ **提供上下文**
> "上次会话完成了问卷填写功能，代码在 packages/frontend/src/views/constitution/QuizView.vue"

✅ **分步骤确认**
> "第一步先改后端接口，你确认一下需要改哪些文件"

✅ **及时反馈**
> "这个方案可以，继续" / "不对，我需要的是..."

### 7.2 不说什么（DON'T）

❌ **模糊描述**
> "把这个功能弄好"（哪个功能？怎么算好？）

❌ **跳跃式需求**
> "先做结果页面，对了顺便把问卷的样式也改一下"

❌ **不反馈直接退出**
> AI 给出方案后不说行不行，直接消失

❌ **一次性说太多**
> "今天把体质辨识、耳穴、经络三个模块都做了"

### 7.3 沟通节奏

```
用户：开始做 [任务名]
        ↓
AI：确认需求，列出涉及的文件和步骤
        ↓
用户：确认/修正
        ↓
AI：开始编码
        ↓
AI：编码完成，说明自测方法
        ↓
用户：测试反馈（成功/有问题）
        ↓
AI：修复问题/继续下一个任务
        ↓
... 循环直到会话结束
```

## 第九部分：总结

### 9.1 核心习惯

1. **开发前读文档** — PROGRESS.md 确认进度
2. **一次一模块** — 不跨模块开发
3. **完成立即提交** — Git commit + 更新文档
4. **部署先清旧** — 删除服务器旧文件
5. **会话有记录** — SESSIONS.md 追加历史

### 9.2 沟通模板速用

**开始会话：**
> "会话目标：[xxx]。当前状态：[xxx]。任务拆解：[1.xxx 2.xxx 3.xxx]"

**任务开始：**
> "现在开始做 [任务名]，涉及文件：[xxx]"

**任务完成：**
> "[任务名] 完成，自测通过，准备提交 Git"

**结束会话：**
> "本次完成：[xxx]。Git 提交：[xxx]。下次从 [xxx] 继续"

### 9.3 版本管理速记

```
改代码 → 提交 Git → 部署前打标签 → 清空旧的 → 上传新的 → 验证
```
