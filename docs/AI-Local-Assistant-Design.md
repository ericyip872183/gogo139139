# AI 本地助手 - 软件操作自动化设计方案

> **项目代号**：Panlei Desktop Assistant (PDA)
> **目标**：创建一个 AI 助手，能够理解用户意图，分析软件界面，自动完成操作
> **版本**：v1.0
> **日期**：2026-03-29

---

## 一、需求概述

### 1.1 核心场景

```
用户：帮我打开微信，搜索「老板」，发送「收到」

AI 助手：
1. 📸 截取屏幕
2. 🔍 识别微信图标，点击打开
3. 🔍 识别搜索框，输入「老板」
4. 🔍 识别联系人，点击进入
5. 🔍 识别输入框，输入「收到」
6. 🔍 识别发送按钮，点击发送
7. ✅ 任务完成
```

### 1.2 功能特性

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 屏幕捕获 | 截取全屏/窗口/区域 | P0 |
| UI 元素识别 | 识别按钮/输入框/菜单等 | P0 |
| 意图理解 | 理解用户的自然语言指令 | P0 |
| 操作执行 | 鼠标点击/拖拽/输入/快捷键 | P0 |
| 结果验证 | 确认操作是否成功 | P1 |
| 多步骤流程 | 支持复杂的多步骤操作 | P1 |
| 学习能力 | 记住常见操作流程 | P2 |

### 2.2 技术选型

| 层级 | 技术选项 | 理由 |
|------|----------|------|
| **桌面框架** | Tauri (Rust + Web) | 轻量、安全、跨平台 |
| **前端** | Vue 3 + TypeScript | 与现有项目一致 |
| **屏幕捕获** | Windows: winapi / screenshots-rs | 高效、性能好 |
| **UI 识别** | GPT-4V / Claude Vision / 豆包多模态 | 强大的视觉理解能力 |

| **操作执行** | enigo (Rust) / nutype | 跨平台鼠标键盘控制 |
| **OCR 备用** | tesseract / easyocr | 文字识别备选方案 |

### 2.3 数据流

```
1. 用户输入：「打开微信，搜索老板，发送收到」
           │
           ▼
2. 意图解析：{"action": "send_message", "app": "wechat", "contact": "老板", "message": "收到"}
           │
           ▼
3. 屏幕捕获：截取当前桌面/窗口
           │
           ▼
4. 视觉理解：识别界面元素位置
           │
           ▼
5. 操作规划：[click(微信), input(搜索框,老板), click(联系人), input(输入框,收到), click(发送)]
           │
           ▼
6. 执行操作：按步骤执行，每步后截图验证
           │
           ▼
7. 结果反馈：「已完成：打开微信，搜索老板，发送『收到』」
```

## 三、核心模块设计

### 3.1 屏幕捕获模块

```rust
// 伪代码：屏幕捕获
mod screen {
    // 截取整个屏幕
    pub fn capture_fullscreen() -> Image

    // 截取指定窗口
    pub fn capture_window(hwnd: isize) -> Image

    // 截取指定区域
    pub fn capture_region(x: i32, y: i32, width: u32, height: u32) -> Image
}
```

**功能**：
- 全屏截取
- 窗口截取（通过窗口标题/句柄）
- 区域截取
- 多显示器支持

### 3.2 UI 元素识别模块

**方案 A：纯视觉模型（推荐）**

```
输入：屏幕截图
输出：
{
  "elements": [
    {"type": "button", "text": "发送", "bbox": [100, 200, 50, 30]},
    {"type": "input", "text": "", "bbox": [50, 100, 300, 40]},
    {"type": "icon", "name": "search", "bbox": [10, 10, 30, 30]}
  ]
}
```

**方案 B：视觉 + OCR 混合**

```
1. 视觉模型识别大区域
2. OCR 识别文字内容
3. 元素分类器识别类型
```

### 3.3 意图解析模块

```typescript
// 输入
用户: "打开微信，搜索老板，发送收到"

// 输出
{
  "task": "send_message",
  "steps": [
    {"action": "open_app", "name": "微信"},
    {"action": "click", "target": "search_input"},
    {"action": "input", "text": "老板"},
    {"action": "click", "target": "contact_item"},
    {"action": "input", "text": "收到"},
    {"action": "click", "target": "send_button"}
  ]
}
```

**关键能力**：
- 任务拆解：将复杂指令拆分为步骤
- 模糊匹配：「发微信」→ 打开微信应用
- 上下文记忆：记住上一步的结果

### 3.4 操作执行模块

```rust
// 伪代码：操作执行
mod actions {
    // 鼠标操作
    pub fn click(x: i32, y: i32)
    pub fn double_click(x: i32, y: i32)
    pub fn right_click(x: i32, y: i32)
    pub fn drag(from: (i32,i32), to: (i32,i32))

    // 键盘操作
    pub fn input_text(text: &str)
    pub fn press_key(key: Key)
    pub fn hotkey(keys: &[Key])

    // 窗口操作
    pub fn activate_window(title: &str)
    pub fn minimize_window(hwnd: isize)
    pub fn close_window(hwnd: isize)
}
```

### 3.5 结果验证模块

每步操作后：
```
1. 截取屏幕
2. 与预期结果对比
3. 成功 → 继续下一步
4. 失败 → 重试 / 报告错误 / 回滚
```

## 五、安全与限制

### 5.1 安全机制

| 机制 | 描述 |
|------|------|
| **操作确认** | 危险操作（删除文件、关闭程序）需用户确认 |
| **权限控制** | 需要用户授权屏幕捕获和自动化权限 |
| **操作日志** | 记录所有操作，支持审计和回滚 |
| **超时保护** | 单次操作超过 30 秒自动停止 |

### 5.2 限制场景

```
❌ 不支持的操作：
- 支付/转账类操作（安全风险）
- 需要验证码/短信验证的操作
- 超出屏幕范围的拖拽
- 涉及敏感信息的操作

⚠️ 需要用户配合：
- 首次启动需要授权
- 部分软件可能需要以管理员运行
```

## 七、关键挑战

### 7.1 技术挑战

1. **UI 元素定位**
   - 软件界面复杂，元素嵌套深
   - 部分自定义控件难以识别
   - 解决方案：视觉模型 + 坐标辅助

2. **操作可靠性**
   - 动画导致元素位置变化
   - 窗口遮挡问题
   - 解决方案：操作前等待 + 重试机制

3. **多步骤流程**
   - 上一步结果影响下一步
   - 需要维护执行上下文
   - 解决方案：每步验证 + 状态机

### 7.2 体验挑战

1. **响应速度**
   - 屏幕截图 + AI 分析耗时
   - 解决方案：增量截图 + 本地轻量模型

2. **理解准确度**
   - 用户指令可能模糊
   - 解决方案：多轮对话确认
