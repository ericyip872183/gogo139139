# Git 版本控制彻底搞懂

> 版本是如何控制的？GitHub 的额外功能到底是什么？

---

## 第一部分：版本是如何控制的？

### 1.1 核心问题：是自动还是手动？

**答案：半自动**

```
Git 会注意到你改了什么（自动）
   ↓
但你要告诉它什么时候存（手动）
   ↓
就像管家注意到你改了文件
   ↓
但你要说"存起来"，它才会存
```

---

### 1.2 Git 的工作机制（人话版）

**Git 的三个区域**：

```
┌─────────────────────────────────────────┐
│  工作区（Working Directory）            │
│  - 你正在改的文件                       │
│  - Git 注意到你在改，但不管             │
│  - 随便改，改坏了也没事                 │
└─────────────────────────────────────────┘
              ↓ git add .
┌─────────────────────────────────────────┐
│  暂存区（Staging Area）                 │
│  - 准备提交的文件                       │
│  - 你说"这些文件我要存起来"             │
│  - 就像把文件放进文件盒                 │
└─────────────────────────────────────────┘
              ↓ git commit
┌─────────────────────────────────────────┐
│  本地仓库（Repository）                 │
│  - 真正存起来的地方                     │
│  - 一个快照，一个版本                   │
│  - 存在 .git/objects/ 里                │
└─────────────────────────────────────────┘
```

---

### 1.3 完整流程（带例子）

**场景**：你改了登录页面的样式

```
第 1 步：改代码（工作区）
├─ 你打开 LoginView.vue
├─ 改了按钮颜色：蓝色 → 红色
├─ Git 看到了，但什么都没说
└─ 这时候还没存！

第 2 步：添加到暂存区（git add）
├─ 你运行：git add .
├─ Git 把改了的文件放进"待提交"列表
└─ 这时候还是没存！

第 3 步：提交到仓库（git commit）
├─ 你运行：git commit -m "改按钮颜色"
├─ Git 说：好的，我存了个快照！
└─ 这时候才真正存起来了！
```

---

### 1.4 你要做什么 vs Git 自动做什么

| 步骤 | 谁来做 | 具体动作 |
|------|--------|---------|
| 注意到你改了什么 | Git 自动 | 它一直在后台看着 |
| 放进暂存区 | 你手动 | git add . |
| 提交到仓库 | 你手动 | git commit -m "描述" |
| 生成版本号 | Git 自动 | 它给每个提交一个哈希值 |
| 打标签 | 你手动 | git tag -a v1.0.0 -m "说明" |

---

### 1.5 类比：管家管理文件

```
你（程序员）在书房改文件
   ↓
管家（Git）在旁边看着
   ↓
你改完了，说：管家，把这些存起来（git add）
   ↓
管家放到文件盒里
   ↓
你再说：管家，存档！（git commit）
   ↓
管家放到档案室，贴个标签"2026-03-22 改按钮颜色"
   ↓
下次想看这个版本，说：管家，把 2026-03-22 的拿出来
   ↓
管家就给你拿出来（git checkout）
```

## 第二部分：代码审查（Pull Request）

### 2.1 技术解释

**Pull Request（PR）**：别人改了你的代码，你可以看看再决定要不要合并

```
流程：
1. 同事 Fork 了你的项目
2. 他改了一个功能
3. 他发起 Pull Request："老板，我改了代码，你看看要不要？"
4. 你在 GitHub 上看到他的修改
5. 你可以评论、提建议
6. 你觉得没问题，点"Merge"合并
```

---

### 2.2 人话比喻

**比喻：投稿编辑**

```
你开了个杂志社（项目）
   ↓
同事写了篇文章（改了代码）
   ↓
投稿给你（Pull Request）
   ↓
你当编辑，看看写得好不好（代码审查）
   ↓
写得不好：退稿，让他改（Comment 提意见）
   ↓
写得好：发表，合并到杂志里（Merge）
```

---

### 2.3 实际例子

**场景**：同事要改登录按钮的颜色

```
同事的操作：
1. 在你的 GitHub 项目页面，点"Fork"（复制一份）
2. 在他电脑上改代码：蓝色 → 红色
3. 提交到他的 GitHub
4. 在你的项目页面，点"New Pull Request"
5. 写说明："改了按钮颜色，更好看"

你的操作：
1. 收到通知：有人发起 PR
2. 点开看：他改了哪里
3. 测试一下：红色好不好看
4. 没问题，点"Merge pull request"
5. 他的修改就合并到你的项目了！
```

---

### 2.4 为什么需要代码审查？

| 原因 | 说人话 |
|------|--------|
| 防止烂代码 | 别让菜鸟把项目搞砸 |
| 统一风格 | 大家写得像一个人写的 |
| 发现 Bug | 多一双眼睛，少一个 Bug |
| 学习进步 | 新人可以学到老人的写法 |

## 第三部分：问题跟踪（Issues）

### 3.1 技术解释

**Issues**：GitHub 自带的问题跟踪系统

```
功能：
├─ 提 Bug："这里有问题"
├─ 提需求："想要这个功能"
├─ 分配任务："小王，你修这个 Bug"
├─ 标签分类："Bug"、"功能请求"、"紧急"
└─ 进度跟踪："进行中"、"已完成"
```

---

### 3.2 人话比喻

**比喻：餐厅的意见本 + 任务分配板**

```
顾客（用户）来吃饭
   ↓
发现问题：菜太咸了
   ↓
写在意见本上（提 Issue）
   ↓
老板（项目维护者）看到
   ↓
写在任务板上："厨师，菜太淡了"（分配任务）
   ↓
厨师修好了，打勾（关闭 Issue）
   ↓
老板告诉顾客："修好了"（Comment 回复）
```

---

### 3.3 实际例子

**场景**：用户发现登录有问题

```
用户操作：
1. 打开项目的 Issues 页面
2. 点"New Issue"
3. 写标题："登录按钮点了没反应"
4. 写详情：
   - 什么浏览器：Chrome
   - 什么系统：Windows 11
   - 怎么复现：点登录按钮，没反应
5. 提交

维护者操作：
1. 收到通知：有新 Issue
2. 看看是不是 Bug：确实是
3. 加标签："Bug"、"紧急"
4. 指派给小王："@小王 你看看"
5. 小王修好了，Comment："修好了，测试一下"
6. 用户测试，没问题，关闭 Issue
```

---

### 3.4 Issues 能干什么？

| 用途 | 说人话 |
|------|--------|
| 报 Bug | "这里有问题，快修" |
| 提需求 | "想要这个功能" |
| 分配任务 | "@小王 你来做" |
| 跟踪进度 | "进行中"、"已完成" |
| 讨论问题 | 大家在 Issue 里评论 |

## 第四部分：自动化构建（Actions）

### 4.1 技术解释

**GitHub Actions**：代码提交后自动执行任务

```
触发条件：
├─ 代码提交（push）
├─ Pull Request 发起
├─ 定时任务（每天/每周）
└─ 手动触发

执行动作：
├─ 自动构建（pnpm run build）
├─ 自动测试（pnpm test）
├─ 自动部署（上传到服务器）
└─ 自动通知（发邮件/钉钉）
```

---

### 4.2 人话比喻

**比喻：自动化流水线**

```
传统方式（没有 Actions）：
你提交代码 → 手动构建 → 手动测试 → 手动部署
   ↓
累死了，还容易出错

有 Actions：
你提交代码 → 自动构建 → 自动测试 → 自动部署
   ↓
你喝茶就行了，机器帮你干
```

---

### 4.3 实际例子

**场景**：每次提交代码后自动部署

```
配置文件（.github/workflows/deploy.yml）：
name: 自动部署
on:
  push:
    branches: [main]  # 每次推送到 main 分支
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3  # 拉代码
      - run: pnpm install          # 安装依赖
      - run: pnpm run build        # 构建
      - run: ./deploy/sync-to-server.sh  # 部署
```

**效果**：

```
你在家提交代码
   ↓
git push
   ↓
GitHub 收到代码
   ↓
自动启动 Actions
   ↓
自动安装依赖
   ↓
自动构建
   ↓
自动部署到服务器
   ↓
你收到邮件："部署成功！"
   ↓
你在喝茶，啥也没干
```

---

### 4.4 Actions 能干什么？

| 用途 | 说人话 |
|------|--------|
| 自动构建 | 提交代码后自动打包 |
| 自动测试 | 提交代码后自动跑测试，有问题马上知道 |
| 自动部署 | 测试通过后自动上线 |
| 自动发通知 | 部署成功/失败发邮件/钉钉 |
| 定时任务 | 每天凌晨自动备份 |

## 第五部分：在线协作

### 5.1 技术解释

**GitHub 在线协作功能**：

```
功能：
├─ 多人编辑同一个文件（在线改）
├─ 实时评论（在代码旁边评论）
├─ 代码建议（别人可以直接帮你改）
└─ 合并冲突解决（网页上解决冲突）
```

---

### 5.2 人话比喻

**比喻：在线文档（腾讯文档/石墨文档）**

```
传统方式：
你改完发给我 → 我改完发给你 → 冲突了，不知道用谁的
   ↓
累死了，还容易乱

在线协作：
我们一起打开同一个文档
   ↓
你改你的，我改我的
   ↓
实时看到对方改了啥
   ↓
有冲突当场解决
```

---

### 5.3 实际例子

**场景**：3 个人一起改一个文件

```
传统 Git 流程（没有 GitHub）：
小王改了 LoginView.vue
   ↓
小李也改了 LoginView.vue
   ↓
两个人都提交
   ↓
冲突了！Git 说：合并冲突
   ↓
三个人开会，决定用谁的
   ↓
累死了

GitHub 在线协作：
小王在网页上改 LoginView.vue
   ↓
小李也在网页上改同一个文件
   ↓
GitHub 说：这里有冲突，你们看看
   ↓
三个人在网页上直接改
   ↓
当场解决，合并
   ↓
轻松
```

## 第六部分：总结

### 6.1 版本控制一句话

> **Git 会注意到你改了什么（自动），但你要告诉它什么时候存（手动）**
>
> - 工作区：你随便改
> - 暂存区：git add 放进去
> - 仓库：git commit 存起来

---

### 6.2 代码审查一句话

> **Pull Request = 投稿编辑**
>
> - 别人改你的代码，你可以看看再决定要不要
> - 可以评论、提建议、让他改
> - 没问题就合并

---

### 6.3 问题跟踪一句话

> **Issues = 意见本 + 任务分配板**
>
> - 用户可以报 Bug、提需求
> - 维护者分配任务、跟踪进度
> - 修好了关闭

---

### 6.4 自动化构建一句话

> **GitHub Actions = 自动化流水线**
>
> - 代码提交后自动构建、测试、部署
> - 你喝茶，机器干活
> - 有问题马上知道

---

### 6.5 在线协作一句话

> **GitHub 在线协作 = 在线文档**
>
> - 多人同时编辑一个文件
> - 实时看到对方改了啥
> - 有冲突当场解决

---
