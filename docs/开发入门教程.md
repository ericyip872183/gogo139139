## 第三部分：关键概念解释（说人话版）

### 3.1 前端 vs 后端

**比喻**：餐厅的前厅和后厨

| 前端（front-end） | 后端（back-end） |
|-----------------|----------------|
| 客人看到的装修 | 客人看不见的厨房 |
| 菜单、桌椅、灯光 | 厨师、锅碗瓢盆 |
| 负责好看、好用 | 负责好吃、卫生 |
| Vue.js 编写 | NestJS 编写 |
| 浏览器运行 | 服务器运行 |

**关系**：
```
客人（用户）
   ↓
前厅（前端）：请问您要什么？
   ↓ 点菜单（API 请求）
后厨（后端）：收到，马上做
   ↓ 端上来（API 响应）
前厅（前端）：您的菜来了
```

### 3.2 源代码 vs 构建产物

**比喻**：食材 vs 成品菜

```
源代码（src/）          构建（build）         构建产物（dist/）
┌─────────────┐        ┌─────────┐        ┌─────────────┐
│ 面粉 + 水    │        │         │        │ 面包        │
│ 蔬菜 + 肉    │  ──→   │  厨师   │  ──→   │ 炒菜        │
│ 设计稿       │        │  机器   │        │ 装修好的店  │
└─────────────┘        └─────────┘        └─────────────┘
   给人看的              加工处理             给客人看的
```

**关键理解**：
- 你改的是 `src/` 源代码（食材）
- 客人用的是 `dist/` 构建产物（成品菜）
- **改了源代码必须运行 `pnpm run build`（重新炒菜）**

> ⚠️ **血泪教训**：上次改了"盼蕾"变"若容"，但没重新构建，服务器还是旧名字！

### 3.3 什么是环境（Environment）

**比喻**：厨房的设备和调料配置

| 环境 | 比喻 | 用途 |
|------|------|------|
| 开发环境（dev） | 试验厨房 | 程序员本地调试，随便改 |
| 生产环境（prod） | 正式营业的厨房 | 客人真正使用的，不能乱动 |

**为什么需要区分？**
- 开发环境：可以出错，改了立即看到效果
- 生产环境：必须稳定，改了要谨慎

### 3.4 什么是容器（Docker）

**比喻**：集装箱

**问题**：
- 你家厨房的调料放在左边
- 我家厨房的调料放在右边
- 换个厨房就不会做菜了 😰

**解决方案**：集装箱

```
┌─────────────────────────────────────┐
│  Docker 集装箱                        │
│  ┌─────┐  ┌─────┐  ┌─────────────┐  │
│  │ 代码 │  │ 工具 │  │ 配置文件    │  │
│  └─────┘  └─────┘  └─────────────┘  │
└─────────────────────────────────────┘
        ↓ 吊起就走
无论放到哪里（任何服务器），打开就能用！
```

**Docker 的好处**：
- 在我电脑上能跑，在你电脑上也能跑
- 不怕"缺少依赖"、"版本不对"
- 一键部署，不用手动安装

### 3.5 什么是数据库

**比喻**：仓库货架

```
数据库（MySQL）
├── User 表 ← 用户信息货架
│   ├── 张三 | admin | 密码 (加密) | 正常
│   └── 李四 | student01 | 密码 | 正常
│
├── Question 表 ← 题库货架
│   ├── 题目 1 | 单选题 | 阴阳是...
│   └── 题目 2 | 判断题 | 舌红苔黄为热...
│
└── Score 表 ← 成绩货架
    ├── 张三 | 考试 1 | 85 分
    └── 李四 | 考试 1 | 92 分
```

**Prisma 是什么？**
- 仓库管理员
- 帮你用代码操作数据库（不用写 SQL）

### 3.6 什么是 API

**比喻**：点菜单

```
前端（服务员）              后端（厨师）
   ↓ 点菜单（API 请求）         ↓
   │ GET /api/questions       │
   │ 我要题库                 │
   │                         │
   │ ← {code:200,            │
   │     data:[...题目...]}   │
   │ 返回题库数据             │
```

**API 格式规范**：
```json
{
  "code": 200,      // 状态码：200=成功，401=没登录，500=出错了
  "message": "成功", // 提示信息
  "data": {...},    // 实际数据
  "timestamp": 1234567890  // 时间戳
}
```

### 3.7 什么是 JWT

**比喻**：游乐园手环

```
登录成功
   ↓
发给你一个手环（Token）
   ↓
玩每个项目都伸手亮手环
   ↓
工作人员检查手环 → 放行
```

**JWT_SECRET 是什么？**
- 是制作手环的模具
- 生成手环用模具 A，验证用模具 B → 对不上 ❌
- **服务器上的模具必须一致**

> ⚠️ **血泪教训**：上次 JWT_SECRET 不一致，登录一直失败！

### 3.8 什么是 CORS

**比喻**：小区门禁

```
浏览器：我要访问 https://pl.7025700.vip/api/login
   ↓
浏览器检查：这个请求是跨域的，需要服务器允许
   ↓
服务器响应头包含：
  Access-Control-Allow-Origin: https://pl.7025700.vip ✅
   ↓
浏览器：好，服务器同意了，放行

如果服务器没返回这个头：
   ↓
浏览器：服务器没同意，拦截 ❌
```

> ⚠️ **血泪教训**：上次 nginx 没配 CORS，浏览器一直拦截请求！

### 3.9 什么是 Nginx

**比喻**：门卫 + 调度员

```
互联网请求
   ↓
┌─────────────────┐
│     Nginx       │ ← 门卫
└─────────────────┘
   ↓         ↓
前端静态文件   后端 API
(/var/www/)   (localhost:3002)
```

**Nginx 负责**：
1. HTTPS 加密（SSL 证书）
2. 静态文件服务（HTML、CSS、JS）
3. API 请求转发（代理到后端）
4. CORS 头添加（门禁卡）

## 第四部分：版本管理（到底在管什么）

### 4.1 为什么需要版本管理

**问题场景**：
```
第 1 天：项目上线了
第 2 天：改了个 Bug
第 3 天：又加了个功能
第 4 天：改错了，想回到第 1 天...
   ↓
没有版本管理：回不去 😭
有版本管理：git checkout 第 1 天 ✅
```

### 4.2 Git 是什么

**比喻**：时间机器 + 保险箱

```
Git 仓库
├── 能回到过去（历史记录）
├── 能保存快照（提交）
├── 能贴标签（版本）
└── 能对比差异（diff）
```

### 4.3 版本管理流程

```
┌─────────────────────────────────────────┐
│ 1. 开发前                                │
│    git status → 确认干净                 │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 2. 开发功能                              │
│    改代码...                             │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 3. 本地构建测试                          │
│    pnpm run build                        │
│    确认没问题                            │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 4. 提交代码                              │
│    git add .                            │
│    git commit -m "描述本次修改"          │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 5. 部署到服务器                          │
│    ./deploy/sync-to-server.sh           │
│    先清旧文件 + 上传新文件                │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 6. 打版本标签                            │
│    git tag -a v1.2.0 -m "版本说明"       │
│    方便以后追溯                          │
└─────────────────────────────────────────┘
```

### 4.4 版本号命名

```
v1.2.3
│ │ │
│ │ └─ 修订号（Bug 修复，小改动）
│ └─── 次版本号（新功能，向后兼容）
└───── 主版本号（大改动，可能不兼容）
```

| 版本号 | 含义 | 示例 |
|--------|------|------|
| v1.0.0 | 首次上线 | 平台正式发布 |
| v1.0.1 | 修了个小 Bug | 修复登录按钮颜色 |
| v1.1.0 | 加了新功能 | 新增题库管理 |
| v2.0.0 | 大改版 | 架构重构 |

### 4.5 如何避免旧文件堆积

**问题根源**：
```
Vite 构建原理：
第 1 次构建 → index-ABC.js
第 2 次构建 → index-DEF.js
第 3 次构建 → index-GHI.js
   ↓
不删除旧的 → 服务器上堆积几十上百个文件！
```

**解决方案**：
```bash
# 部署脚本里已经包含
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"
```

**每次上传前先清空，确保只有当前版本**

## 第八部分：环境问题排查

### 8.1 常见问题速查

| 现象 | 可能原因 | 解决方法 |
|------|----------|----------|
| 页面显示旧内容 | 浏览器缓存 | Ctrl+Shift+R 强制刷新 |
| 登录返回 401 | JWT_SECRET 不一致 | 检查服务器.env 文件 |
| API 请求失败 | CORS 配置缺失 | 检查 nginx 配置 |
| 功能部分正常 | 前后端版本不匹配 | 重新运行同步脚本 |
| 服务无法访问 | 后端进程挂了 | pm2 restart |
| 端口被占用 | 另一个服务在用 | netstat -tlnp 查占用 |

### 8.2 排查步骤

```
1. 确认问题现象
   ↓
2. 检查服务器文件（SSH 登录查看）
   ↓
3. 检查服务状态（pm2 status）
   ↓
4. 检查日志（pm2 logs）
   ↓
5. 测试 API（curl 直接调用）
   ↓
6. 定位问题并修复
   ↓
7. 重新部署验证
```

## 第九部分：真实案例复盘 — "登录失效"风波

**时间**：2026-03-21

**起因**：用户要求把项目品牌名从"盼蕾"改为"若容"

**初始状态**：项目运行正常，用户访问 https://pl.7025700.vip

### 9.2 问题是如何发生的（一步步走偏）

```
第 1 步：改源代码 ✅
├─ 修改 LoginView.vue：盼蕾 → 若容
├─ 修改 index.html：盼蕾 → 若容

   └─ 以为：改了代码就生效

├─ 运行部署脚本
├─ 问题：没有清空服务器旧文件
└─ 结果：新旧文件混在一起

第 3 步：用户反馈还是"盼蕾" ❌
├─ 原因：dist/ 没重新构建
└─ 浏览器读的是旧 index.html

第 4 步：重新构建上传 ✅
├─ 运行 pnpm run build:frontend
├─ 但：服务器上堆积了大量旧 JS 文件

├─ 前端有 lookupByUsername 接口调用

├─ 浏览器请求被拦截
└─ nginx 没有配置 CORS 头

第 9 步：全部修复完成 ✅
```

### 9.4 AI 是如何被"带偏"的

**AI 的思维过程**：

```
用户：登录失效了

AI 第 1 轮思考：
├─ 假设 1：Token 过期了？
├─ 检查：JWT 有效期正常 ✅
└─ 结论：不是这个问题

AI 第 2 轮思考：
├─ 假设 2：后端 API 有问题？
├─ 测试：curl 调用登录 API
└─ 结果：返回 token ✅ — 说明后端正常

AI 第 3 轮思考：
├─ 矛盾：curl 正常，浏览器失败
├─ 假设 3：浏览器缓存？
├─ 建议：清除缓存/无痕模式
└─ 结果：还是失败 ❌

AI 第 4 轮思考：
├─ 关键线索：curl 能拿到 token，但浏览器说 401
├─ 洞察：同一个后端，为什么两种结果？
├─ 假设 4：CORS 问题！
├─ 检查：nginx 配置
└─ 发现：没有 Access-Control-Allow-Origin ❌

AI 第 5 轮思考：
├─ 验证：手动添加 CORS 头
├─ 测试：浏览器登录
└─ 结果：成功 ✅
```

**为什么会被带偏？**

| 轮次 | AI 怀疑的方向 | 为什么错 |
|------|--------------|---------|
| 第 1 轮 | Token 过期 | 只看了配置，没看实际验证 |
| 第 2 轮 | 后端 API 问题 | curl 测试成功，排除 |
| 第 3 轮 | 浏览器缓存 | 常见原因，但这次不是 |
| 第 4 轮 | CORS 问题 | 命中！ curl 和浏览器行为不同 |

**关键点**：
- AI 不是全知全能，是靠**排除法**一步步缩小范围
- 每次测试排除一个可能性
- 直到找到真正的根因

### 9.6 如何清理旧文件

**问题**：服务器上有数十个旧 JS/CSS 文件

**原因**：Vite 构建原理

```
第 1 次构建：index-Ck6_hncp.js  (哈希值 hncp)
第 2 次构建：index-D7xQM2z.js   (哈希值 D7xQM2z)
第 3 次构建：index-Bv9Kp3L.js   (哈希值 Bv9Kp3L)
...
```

**为什么文件名变化？**
- Vite 自动给文件加哈希值
- 内容变了 → 哈希值变 → 文件名变
- 好处：浏览器缓存不会冲突
- 坏处：不删除旧文件会堆积

**解决方案**：部署脚本已修复

```bash
# 上传前先清空
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"

# 再上传新文件
scp -r packages/frontend/dist/* root@120.77.144.5:/var/www/panlei/
```

## 第十一部分：常见疑问解答

### 11.1 为什么 AI 会"忘记"构建？

**用户的疑问**：
> "你不构建根本就运行不了，我也不可能看到结果，但每次修改后我都能看到结果"

**真相揭秘**：

```
┌─────────────────────────────────────────────────────────┐
│ 本地开发模式（pnpm dev）                                │
├─────────────────────────────────────────────────────────┤
│ 你改代码 → Vite 自动实时编译 → 浏览器自动刷新           │
│ 所以：你每次修改都能立即看到结果 ✅                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 生产部署模式（pnpm run build）                          │
├─────────────────────────────────────────────────────────┤
│ 你改代码 → 必须手动运行 build → 生成新的 dist/          │
│ 否则：上传的还是旧的 dist/ ❌                           │
└─────────────────────────────────────────────────────────┘
```

**为什么会有这个矛盾？**

| 场景 | 构建方式 | 文件名 | 是否需要手动执行 |
|------|---------|--------|-----------------|
| 本地开发（pnpm dev） | Vite 自动实时构建 | 不变 | 不需要 |
| 生产部署（pnpm run build） | 手动执行 | 带哈希 | 需要 |

**AI 犯错的过程**：
```
1. AI 用 pnpm dev 测试 → 自动更新 → 看到效果 ✅
2. AI 以为 dist/ 已经是最新的 → 其实不是 ❌
3. 上传旧 dist/ 到服务器 → 显示旧内容 ❌
```

**根本原因**：
- `pnpm dev` 模式的 dist/ 和 `pnpm run build` 的 dist/ 是**两回事**
- 开发模式：代码不压缩，方便调试
- 生产模式：代码压缩合并，优化性能

---

### 11.2 构建（Build）到底是什么？

**比喻**：装修房子

```
源代码（src/）              构建（Build）            构建产物（dist/）
┌─────────────────┐        ┌─────────┐        ┌─────────────────┐
│ 设计图纸         │        │         │        │ 装修好的房子     │
│ 木材 + 油漆      │  ──→   │  施工队 │  ──→   │ 可以入住         │
│ 原始材料         │        │  机器   │        │ 给用户使用       │
└─────────────────┘        └─────────┘        └─────────────────┘
   给人看的                  加工处理               给客人住的
```

**具体变化**：

你写的代码（源代码）：
```vue
<!-- LoginView.vue -->
<template>
  <div class="login-page">
    <h1>若容虚拟数据平台</h1>
  </div>
</template>

<script lang="ts">
// TypeScript 代码，浏览器看不懂
</script>
```

构建后（dist/）：
```html
<!-- index.html -->
<!DOCTYPE html>
<html>
<head>
  <title>若容虚拟数据平台</title>
  <script src="/assets/index-Ck6_hncp.js"></script>
  <!-- 机器生成的文件，人看不懂但浏览器能跑 -->
</html>
```

**构建过程做了 4 件事**：

| 步骤 | 做了什么 | 比喻 |
|------|---------|------|
| 1. 翻译 | TypeScript → JavaScript | 中文翻译成英文 |
| 2. 合并 | 100 个小文件 → 3 个大文件 | 零散食材打包成套餐 |
| 3. 压缩 | 去掉空格/注释，变量名缩短 | 真空压缩，减小体积 |
| 4. 加哈希 | 文件名加上内容指纹 | 给产品贴防伪标签 |

### 11.3 为什么改了代码必须重新构建？

**错误做法**：
```
改了 LoginView.vue（源代码）
   ↓
直接上传到服务器（没构建）
   ↓
服务器还是旧的 dist/
   ↓
用户看到的还是旧页面 ❌
```

**正确做法**：
```
改了 LoginView.vue（源代码）
   ↓
运行 pnpm run build:frontend（重新构建）
   ↓
生成新的 dist/（包含最新修改）
   ↓
上传新的 dist/ 到服务器
   ↓
用户看到新页面 ✅
```

### 13.4 为什么这个项目可以有网址，有的不行？

**网址访问需要什么？完整链条**：

```
1. 有公网 IP 地址
   ↓
2. 有域名（DNS 解析到 IP）
   ↓
3. 服务器有端口开放（80/443）
   ↓
4. 有 Nginx 处理请求
   ↓
5. 有 SSL 证书（HTTPS）
   ↓
6. 后端服务在运行
   ↓
✅ 可以访问！
```

**这个项目为什么可以有网址**：

| 条件 | 状态 | 说明 |
|------|------|------|
| 公网 IP | ✅ 有 | 120.77.144.5（阿里云服务器） |
| 域名 | ✅ 有 | pl.7025700.vip |
| 端口开放 | ✅ 有 | 80/443 端口已开放 |
| Nginx | ✅ 有 | panlei-frontend 容器里的 Nginx |
| SSL 证书 | ✅ 有 | Let's Encrypt 免费证书 |
| 服务运行 | ✅ 有 | 4 个容器都在运行 |

**有的项目为什么不行**：

| 原因 | 说人话 | 解决方法 |
|------|--------|---------|
| 没有公网 IP | 服务器在你家，别人找不到 | 买阿里云/腾讯云服务器 |
| 没有域名 | 只有 IP，没有网址 | 买个域名（几十块钱一年） |
| 端口没开放 | 防火墙挡住了 | 在云服务器控制台开放 80/443 |
| 没有 Nginx | 直接跑后端，不安全 | 加 Nginx 做反向代理 |
| 没有 SSL 证书 | 浏览器提示不安全 | 申请 Let's Encrypt 免费证书 |
| 服务没跑起来 | 容器没启动 | docker-compose up -d |

### 13.5 用比喻理解网址访问

**比喻：开餐厅**

| 技术概念 | 餐厅比喻 |

## 第十四部分：本地开发 vs 生产部署

### 14.1 为什么要分开？

**比喻**：试验厨房 vs 正式营业的餐厅

```
试验厨房（本地开发）
├─ 可以随便试菜，出错没关系
├─ 客人看不到
└─ 只有厨师自己用

正式营业餐厅（生产部署）
├─ 客人来吃饭的地方
├─ 必须稳定，不能出错
└─ 客人能看到
```

**分开的好处**：

| 原因 | 说人话 |
|------|--------|
| 开发可以随便改 | 试验厨房随便试新菜 |
| 生产必须稳定 | 正式餐厅不能随便改 |
| 开发改坏了不影响用户 | 试验厨房炸了不影响客人 |
| 可以快速迭代 | 试验厨房快速试菜 |

**本地开发 vs 生产部署对比**：

| 方面 | 本地开发（你的电脑） | 生产部署（服务器） |
|------|-------------------|------------------|
| 目的 | 程序员写代码、调试 | 真实用户使用 |
| 数据库 | MySQL（Docker） | MySQL（Docker） |
| 前端运行 | pnpm dev（端口 5173） | Nginx（端口 80/443） |
| 后端运行 | pnpm dev（端口 3002） | 容器化（端口 3000） |
| 数据 | 可以随便删改 | 必须备份、不能删 |
| 访问地址 | http://localhost:5173 | https://pl.7025700.vip |

### 14.2 如何保持同步？

**比喻**：试验厨房试好菜，好了再端到正式餐厅

```
本地开发
   ↓
试菜（测试）
   ↓
觉得没问题
   ↓
打包装车（Git 提交）
   ↓
端到正式餐厅（部署到服务器）
   ↓
客人品尝（用户访问）
```

**同步流程**：
```
本地开发 → 本地测试 → Git 提交 → 部署到服务器 → 验证
```

**什么时候同步最合理？**

| 时机 | 说人话 |
|------|--------|
| 功能开发完成 | 新菜做好了，尝过没问题 |
| Bug 修复完成 | 菜做坏了，重新做好了 |
| 小功能改完 | 菜加了点调料，没问题 |

**什么时候不要同步？**

| 时机 | 说人话 | 为什么 |
|------|--------|--------|
| 功能开发到一半 | 菜做了一半，还没熟 | 客人吃了会生病 |
| Bug 还没测试 | 菜做坏了，还没修好 | 客人吃了会拉肚子 |
| 改了一半又改回来 | 菜做了又改，还没定 | 客人不知道吃什么 |

**完整同步流程**：

```
本地开发（试验厨房）
   ↓
本地测试（尝菜）
   ↓Git 提交（打包）
   ↓部署到服务器（端到正式餐厅）
   ↓服务器上验证（再尝一遍）
   ↓用户使用（客人吃）
```

**Git 同步的具体操作**：

```bash

# 2. 本地测试没问题
# 3. 提交 Git

git commit -m "feat: 新功能完成"
git tag -a v1.1.8 -m "新功能版本"

# 4. 部署到服务器
./deploy/sync-to-server.sh

# 5. 服务器上验证
curl https://pl.7025700.vip | grep title
```

## 第十六部分：Git 版本控制详解

### 16.1 版本是如何控制的？自动还是手动？

**答案：半自动**

```
Git 会注意到你改了什么（自动）
   ↓
但你要告诉它什么时候存（手动）
   ↓
就像管家注意到你改了文件
   ↓
但你要说"存起来"，它才会存
```

**Git 的三个区域**：

```
工作区（你正在改的文件）
   ↓ git add .
暂存区（准备提交的文件）
   ↓ git commit
本地仓库（真正存起来）
```

**你要做什么 vs Git 自动做什么**：

| 步骤 | 谁来做 | 具体动作 |
|------|--------|---------|
| 注意到你改了什么 | Git 自动 | 它一直在后台看着 |
| 放进暂存区 | 你手动 | git add . |
| 提交到仓库 | 你手动 | git commit -m "描述" |
| 生成版本号 | Git 自动 | 它给每个提交一个哈希值 |
| 打标签 | 你手动 | git tag -a v1.0.0 -m "说明" |

### 16.2 代码审查（Pull Request）

```
写得不好：退稿，让他改（Comment 提意见）

写得好：发表，合并到杂志里（Merge）
```

**流程**：
1. 同事 Fork 了你的项目

## 第十七部分：git status 详解

### 17.1 git status 什么时候用？

**核心答案**：是你主动用，Git 不会自己用

```
你："现在看看改了什么"
   ↓
你说：git status
   ↓
Git 输出：LoginView.vue 变了
   ↓
你看到了

Git 不会主动跳出来告诉你："你改了文件！"
```

**常用时机**：

| 时机 | 说什么 | 看到什么 |
|------|--------|---------|
| 开发前 | `git status` | 确认工作区干净 |
| 开发后 | `git status` | 看看改了哪些文件 |
| 提交前 | `git status` | 确认哪些文件要提交 |
| 不确定时 | `git status` | 看看现在什么状态 |

---

### 17.2 实际例子

**场景 1：开发前确认状态**

```
你坐在电脑前，准备开发新功能
   ↓
你先运行：git status
   ↓
Git 输出：
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean
   ↓
你：好的，工作区干净，可以开始开发了
```

**为什么要先看看？**
- 确认上次的工作已经提交
- 确认没有遗漏的文件
- 心里有底

---

**场景 2：开发后看看改了什么**

```
你改了一下午代码
   ↓
你运行：git status
   ↓
Git 输出：
On branch main
Changes not staged for commit:

        modified:   LoginView.vue
        modified:   auth.controller.ts

Untracked files:

        constitution/

no changes added to commit
   ↓
你：哦，我改了 2 个文件，还新建了一个文件夹
```

**输出解读**：

```
modified: LoginView.vue
├─ 意思是：这个文件你改了
└─ 但还没提交！

Untracked files: constitution/
├─ 意思是：这是个新文件/文件夹
└─ Git 以前没见过它
```

---

**场景 3：提交前确认**

```
你准备提交代码了
   ↓
你先运行：git status
   ↓
Git 输出：
Changes to be committed:

        modified:   LoginView.vue
        modified:   auth.controller.ts
        new file:   constitution/QuizView.vue
   ↓
你：好的，这 3 个文件就是要提交的
   ↓
你运行：git commit -m "feat: 新增体质辨识问卷"
```

**为什么要提交前看看？**
- 确认没有遗漏文件
- 确认没有不该提交的文件（比如密码文件）
- 心里有数

---

### 17.3 git status 输出详解

**典型输出**：

```

Your branch is up to date with 'origin/main'.

        modified:   file1.txt

Untracked files:

        newfile.txt

no changes added to commit
```

**逐行解读**：

| 输出 | 说人话 |
|------|--------|
| `On branch main` | 你现在在 main 分支上 |
| `Your branch is up to date` | 你的分支和服务器一样，没有落后 |

| `modified: file1.txt` | file1.txt 被你改了 |
| `Untracked files` | 有些文件 Git 没见过（新文件） |
| `no changes added to commit` | 还没有文件准备提交 |

```
命令行 = 你自己开（手动挡）
   ↓
图形界面 = 自动驾驶（自动挡）
   ↓
但都是同一辆车
```

**指令和软件的关系**：

```
Git 软件 = 汽车本身
   ↓
git status = 看仪表盘
git add = 踩油门
git commit = 挂挡
git push = 开车上路
```

### 17.9 常用的 Git 指令

| 指令 | 说人话 | 什么时候用 |
|------|--------|-----------|
| `git status` | 看看现在什么状态 | 不确定改了什么时 |
| `git add <文件>` | 把这个文件放进暂存区 | 准备提交前 |
| `git commit -m "描述"` | 存到本地仓库 | 功能完成时 |
| `git log --oneline` | 看看历史记录 | 想看以前改了什么 |
| `git push` | 推送到远程仓库 | 要备份或分享时 |

### 18.8 如何回滚到某个版本

**三种方法，看你的目的**：

#### 方法 1：临时查看（不修改代码）

```bash
# 1. 先看看有哪些版本
git log --oneline

# 2. 临时切换到 v1.2.0
git checkout v1.2.0

# 3. 看完后回到最新版
git checkout master
```

**适用**：只是想看看某个版本长什么样

---

#### 方法 2：正式回滚（保留历史）

```bash
# 回滚到 v1.2.0，撤销之后的所有改动
git revert v1.2.1..HEAD

# 或者回滚单个提交
git revert <commit-id>
```

**适用**：正式回滚，但不想删除中间的历史
**好处**：历史记录保留，别人知道你回滚过

---

#### 方法 3：强行回到某个版本（丢弃之后的历史）

```bash
# 硬重置到 v1.2.0
git reset --hard v1.2.0
```

**适用**：确定之后的都不要了
**危险**：v1.2.0 之后的所有提交都没了，慎用！

---

#### 三种方法对比

| 方法 | 适用场景 | 危险性 |
|------|---------|--------|
| `git checkout` | 临时看看 | ✅ 安全 |
| `git revert` | 正式回滚 | ✅ 安全 |
| `git reset --hard` | 丢弃历史 | ⚠️ 危险 |

---

#### 回滚前一定要做

```bash
# 1. 先看看当前有没有未提交的修改
git status

# 2. 有修改的话，先保存
git stash  # 临时保存修改

# 3. 再回滚
```

---

#### 一句话总结

| 你想做什么 | 用什么命令 |
|-----------|-----------|
| 看看某个版本 | `git checkout v1.2.0` |
| 撤销但保留历史 | `git revert <版本>` |
| 彻底回到过去 | `git reset --hard <版本>` |

**提醒**：回滚前最好先跟我说一声，我帮你确认用哪个方法。
