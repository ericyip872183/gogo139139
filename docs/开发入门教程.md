## 第二部分：项目文件结构（用餐厅比喻）

### 2.1 整体结构

想象你开了一家餐厅：

```
餐厅（项目）
├── 厨房（backend/）— 做菜的地方
├── 装修（frontend/）— 客人看到的环境
├── 仓库（node_modules/）— 存放调料和工具
├── 菜单（package.json）— 列出有什么菜
└── 规章制度（.md 文档）— 操作规范
```

```
panlei/  ← 整个餐厅
```

## 第三部分：关键概念解释（说人话版）

### 3.1 前端 vs 后端

**比喻**：餐厅的前厅和后厨

| 前端（front-end） | 后端（back-end） |
|-----------------|----------------|
| 客人看到的装修 | 客人看不见的厨房 |
| 菜单、桌椅、灯光 | 厨师、锅碗瓢盆 |
| 负责好看、好用 | 负责好吃、卫生 |
| Vue.js 编写 | NestJS 编写 |
| 浏览器运行 | 服务器运行 |

**关系**：
```
客人（用户）
   ↓
前厅（前端）：请问您要什么？
   ↓ 点菜单（API 请求）
后厨（后端）：收到，马上做
   ↓ 端上来（API 响应）
前厅（前端）：您的菜来了
```

### 3.2 源代码 vs 构建产物

**比喻**：食材 vs 成品菜

```
源代码（src/）          构建（build）         构建产物（dist/）
┌─────────────┐        ┌─────────┐        ┌─────────────┐
│ 面粉 + 水    │        │         │        │ 面包        │
│ 蔬菜 + 肉    │  ──→   │  厨师   │  ──→   │ 炒菜        │
│ 设计稿       │        │  机器   │        │ 装修好的店  │
└─────────────┘        └─────────┘        └─────────────┘
   给人看的              加工处理             给客人看的
```

**关键理解**：
- 你改的是 `src/` 源代码（食材）
- 客人用的是 `dist/` 构建产物（成品菜）
- **改了源代码必须运行 `pnpm run build`（重新炒菜）**

> ⚠️ **血泪教训**：上次改了"盼蕾"变"若容"，但没重新构建，服务器还是旧名字！

### 3.3 什么是环境（Environment）

**比喻**：厨房的设备和调料配置

| 环境 | 比喻 | 用途 |
|------|------|------|
| 开发环境（dev） | 试验厨房 | 程序员本地调试，随便改 |
| 生产环境（prod） | 正式营业的厨房 | 客人真正使用的，不能乱动 |

**为什么需要区分？**
- 开发环境：可以出错，改了立即看到效果
- 生产环境：必须稳定，改了要谨慎

### 3.4 什么是容器（Docker）

**比喻**：集装箱

**问题**：
- 你家厨房的调料放在左边
- 我家厨房的调料放在右边
- 换个厨房就不会做菜了 😰

**解决方案**：集装箱

```
┌─────────────────────────────────────┐
│  Docker 集装箱                        │
│  ┌─────┐  ┌─────┐  ┌─────────────┐  │
│  │ 代码 │  │ 工具 │  │ 配置文件    │  │
│  └─────┘  └─────┘  └─────────────┘  │
└─────────────────────────────────────┘
        ↓ 吊起就走
无论放到哪里（任何服务器），打开就能用！
```

**Docker 的好处**：
- 在我电脑上能跑，在你电脑上也能跑
- 不怕"缺少依赖"、"版本不对"
- 一键部署，不用手动安装

### 3.5 什么是数据库

**比喻**：仓库货架

```
数据库（MySQL）
├── User 表 ← 用户信息货架
│   ├── 张三 | admin | 密码 (加密) | 正常
│   └── 李四 | student01 | 密码 | 正常
│
├── Question 表 ← 题库货架
│   ├── 题目 1 | 单选题 | 阴阳是...
│   └── 题目 2 | 判断题 | 舌红苔黄为热...
│
└── Score 表 ← 成绩货架
    ├── 张三 | 考试 1 | 85 分
    └── 李四 | 考试 1 | 92 分
```

**Prisma 是什么？**
- 仓库管理员
- 帮你用代码操作数据库（不用写 SQL）

### 3.6 什么是 API

**比喻**：点菜单

```
前端（服务员）              后端（厨师）
   ↓ 点菜单（API 请求）         ↓
   │ GET /api/questions       │
   │ 我要题库                 │
   │                         │
   │ ← {code:200,            │
   │     data:[...题目...]}   │
   │ 返回题库数据             │
```

**API 格式规范**：
```json
{
  "code": 200,      // 状态码：200=成功，401=没登录，500=出错了
  "message": "成功", // 提示信息
  "data": {...},    // 实际数据
  "timestamp": 1234567890  // 时间戳
}
```

### 3.7 什么是 JWT

**比喻**：游乐园手环

```
登录成功
   ↓
发给你一个手环（Token）
   ↓
玩每个项目都伸手亮手环
   ↓
工作人员检查手环 → 放行
```

**JWT_SECRET 是什么？**
- 是制作手环的模具
- 生成手环用模具 A，验证用模具 B → 对不上 ❌
- **服务器上的模具必须一致**

> ⚠️ **血泪教训**：上次 JWT_SECRET 不一致，登录一直失败！

### 3.8 什么是 CORS

**比喻**：小区门禁

```
浏览器：我要访问 https://pl.7025700.vip/api/login
   ↓
浏览器检查：这个请求是跨域的，需要服务器允许
   ↓
服务器响应头包含：
  Access-Control-Allow-Origin: https://pl.7025700.vip ✅
   ↓
浏览器：好，服务器同意了，放行

如果服务器没返回这个头：
   ↓
浏览器：服务器没同意，拦截 ❌
```

> ⚠️ **血泪教训**：上次 nginx 没配 CORS，浏览器一直拦截请求！

### 3.9 什么是 Nginx

**比喻**：门卫 + 调度员

```
互联网请求
   ↓
┌─────────────────┐
│     Nginx       │ ← 门卫
└─────────────────┘
   ↓         ↓
前端静态文件   后端 API
(/var/www/)   (localhost:3002)
```

**Nginx 负责**：
1. HTTPS 加密（SSL 证书）
2. 静态文件服务（HTML、CSS、JS）
3. API 请求转发（代理到后端）
4. CORS 头添加（门禁卡）

## 第四部分：版本管理（到底在管什么）

### 4.1 为什么需要版本管理

**问题场景**：
```
第 1 天：项目上线了
第 2 天：改了个 Bug
第 3 天：又加了个功能
第 4 天：改错了，想回到第 1 天...
   ↓
没有版本管理：回不去 😭
有版本管理：git checkout 第 1 天 ✅
```

### 4.2 Git 是什么

**比喻**：时间机器 + 保险箱

```
Git 仓库
├── 能回到过去（历史记录）
├── 能保存快照（提交）
├── 能贴标签（版本）
└── 能对比差异（diff）
```

### 4.3 版本管理流程

```
┌─────────────────────────────────────────┐
│ 1. 开发前                                │
│    git status → 确认干净                 │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 2. 开发功能                              │
│    改代码...                             │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 3. 本地构建测试                          │
│    pnpm run build                        │
│    确认没问题                            │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 4. 提交代码                              │
│    git add .                            │
│    git commit -m "描述本次修改"          │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 5. 部署到服务器                          │
│    ./deploy/sync-to-server.sh           │
│    先清旧文件 + 上传新文件                │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 6. 打版本标签                            │
│    git tag -a v1.2.0 -m "版本说明"       │
│    方便以后追溯                          │
└─────────────────────────────────────────┘
```

### 4.4 版本号命名

```
v1.2.3
│ │ │
│ │ └─ 修订号（Bug 修复，小改动）
│ └─── 次版本号（新功能，向后兼容）
└───── 主版本号（大改动，可能不兼容）
```

| 版本号 | 含义 | 示例 |
|--------|------|------|
| v1.0.0 | 首次上线 | 平台正式发布 |
| v1.0.1 | 修了个小 Bug | 修复登录按钮颜色 |
| v1.1.0 | 加了新功能 | 新增题库管理 |
| v2.0.0 | 大改版 | 架构重构 |

### 4.5 如何避免旧文件堆积

**问题根源**：
```
Vite 构建原理：
第 1 次构建 → index-ABC.js
第 2 次构建 → index-DEF.js
第 3 次构建 → index-GHI.js
   ↓
不删除旧的 → 服务器上堆积几十上百个文件！
```

**解决方案**：
```bash
# 部署脚本里已经包含
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"
```

**每次上传前先清空，确保只有当前版本**

## 第七部分：文件调用关系图

### 7.1 用户登录流程

```
用户在浏览器输入账号密码
         ↓
packages/frontend/src/views/auth/LoginView.vue
         ↓ 收集表单数据
packages/frontend/src/api/auth.ts
         ↓ 发送 HTTP 请求

         ↓ Nginx 转发
packages/backend/src/modules/auth/auth.controller.ts
         ↓ 验证账号密码
packages/backend/src/modules/auth/auth.service.ts
         ↓ 查询数据库

         ↓ 验证通过，生成 Token
JWT.sign({userId}, JWT_SECRET)
         ↓ 返回给前端
{code:200, data:{token:'xxx', user:{...}}}
         ↓ 存储到 localStorage
前端保存 Token，跳转到首页
```

## 第八部分：环境问题排查

### 8.1 常见问题速查

| 现象 | 可能原因 | 解决方法 |
|------|----------|----------|
| 页面显示旧内容 | 浏览器缓存 | Ctrl+Shift+R 强制刷新 |
| 登录返回 401 | JWT_SECRET 不一致 | 检查服务器.env 文件 |
| API 请求失败 | CORS 配置缺失 | 检查 nginx 配置 |
| 功能部分正常 | 前后端版本不匹配 | 重新运行同步脚本 |
| 服务无法访问 | 后端进程挂了 | pm2 restart |
| 端口被占用 | 另一个服务在用 | netstat -tlnp 查占用 |

### 8.2 排查步骤

```
1. 确认问题现象
   ↓
2. 检查服务器文件（SSH 登录查看）
   ↓
3. 检查服务状态（pm2 status）
   ↓
4. 检查日志（pm2 logs）
   ↓
5. 测试 API（curl 直接调用）
   ↓
6. 定位问题并修复
   ↓
7. 重新部署验证
```

## 第九部分：总结

### 9.1 核心概念一句话

| 概念 | 一句话解释 |
|------|-----------|
| 前端 | 用户看到的界面 |
| 后端 | 处理请求的逻辑 |
| 数据库 | 存储数据的仓库 |
| API | 前后端对话的方式 |
| JWT | 登录后的身份手环 |
| CORS | 浏览器的跨域安全检查 |
| Nginx | 门卫 + 调度员 |
| Docker | 集装箱（环境打包） |
| Git | 时间机器（版本管理） |
| 构建 | 源代码 → 成品（炒菜） |

### 9.2 开发流程一句话

> 读文档 → 改代码 → 本地测试 → 提交 Git → 部署验证

### 9.3 版本管理一句话

> 每次修改都提交，每次部署都打标签

> 每个概念搞清楚了，开发就简单了。
