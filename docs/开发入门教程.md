### 2.1 整体结构

```
餐厅（项目）

├── 菜单（package.json）— 列出有什么菜
```

## 第三部分：关键概念解释（说人话版）

### 3.1 前端 vs 后端

**比喻**：餐厅的前厅和后厨

| 前端（front-end） | 后端（back-end） |
|-----------------|----------------|
| 客人看到的装修 | 客人看不见的厨房 |
| 菜单、桌椅、灯光 | 厨师、锅碗瓢盆 |
| 负责好看、好用 | 负责好吃、卫生 |
| Vue.js 编写 | NestJS 编写 |
| 浏览器运行 | 服务器运行 |

**关系**：
```
客人（用户）
   ↓
前厅（前端）：请问您要什么？
   ↓ 点菜单（API 请求）
后厨（后端）：收到，马上做
   ↓ 端上来（API 响应）
前厅（前端）：您的菜来了
```

### 3.2 源代码 vs 构建产物

**比喻**：食材 vs 成品菜

```
源代码（src/）          构建（build）         构建产物（dist/）
┌─────────────┐        ┌─────────┐        ┌─────────────┐
│ 面粉 + 水    │        │         │        │ 面包        │
│ 蔬菜 + 肉    │  ──→   │  厨师   │  ──→   │ 炒菜        │
│ 设计稿       │        │  机器   │        │ 装修好的店  │
└─────────────┘        └─────────┘        └─────────────┘
   给人看的              加工处理             给客人看的
```

**关键理解**：
- 你改的是 `src/` 源代码（食材）
- 客人用的是 `dist/` 构建产物（成品菜）
- **改了源代码必须运行 `pnpm run build`（重新炒菜）**

> ⚠️ **血泪教训**：上次改了"盼蕾"变"若容"，但没重新构建，服务器还是旧名字！

### 3.3 什么是环境（Environment）

**比喻**：厨房的设备和调料配置

| 环境 | 比喻 | 用途 |
|------|------|------|
| 开发环境（dev） | 试验厨房 | 程序员本地调试，随便改 |
| 生产环境（prod） | 正式营业的厨房 | 客人真正使用的，不能乱动 |

**为什么需要区分？**
- 开发环境：可以出错，改了立即看到效果
- 生产环境：必须稳定，改了要谨慎

### 3.4 什么是容器（Docker）

**比喻**：集装箱

**问题**：
- 你家厨房的调料放在左边
- 我家厨房的调料放在右边
- 换个厨房就不会做菜了 😰

**解决方案**：集装箱

```
┌─────────────────────────────────────┐
│  Docker 集装箱                        │
│  ┌─────┐  ┌─────┐  ┌─────────────┐  │
│  │ 代码 │  │ 工具 │  │ 配置文件    │  │
│  └─────┘  └─────┘  └─────────────┘  │
└─────────────────────────────────────┘
        ↓ 吊起就走
无论放到哪里（任何服务器），打开就能用！
```

**Docker 的好处**：
- 在我电脑上能跑，在你电脑上也能跑
- 不怕"缺少依赖"、"版本不对"
- 一键部署，不用手动安装

### 3.5 什么是数据库

**比喻**：仓库货架

```
数据库（MySQL）
├── User 表 ← 用户信息货架
│   ├── 张三 | admin | 密码 (加密) | 正常
│   └── 李四 | student01 | 密码 | 正常
│
├── Question 表 ← 题库货架
│   ├── 题目 1 | 单选题 | 阴阳是...
│   └── 题目 2 | 判断题 | 舌红苔黄为热...
│
└── Score 表 ← 成绩货架
    ├── 张三 | 考试 1 | 85 分
    └── 李四 | 考试 1 | 92 分
```

**Prisma 是什么？**
- 仓库管理员
- 帮你用代码操作数据库（不用写 SQL）

### 3.6 什么是 API

**比喻**：点菜单

```
前端（服务员）              后端（厨师）
   ↓ 点菜单（API 请求）         ↓
   │ GET /api/questions       │
   │ 我要题库                 │
   │                         │
   │ ← {code:200,            │
   │     data:[...题目...]}   │
   │ 返回题库数据             │
```

**API 格式规范**：
```json
{
  "code": 200,      // 状态码：200=成功，401=没登录，500=出错了
  "message": "成功", // 提示信息
  "data": {...},    // 实际数据
  "timestamp": 1234567890  // 时间戳
}
```

### 3.7 什么是 JWT

**比喻**：游乐园手环

```
登录成功
   ↓
发给你一个手环（Token）
   ↓
玩每个项目都伸手亮手环
   ↓
工作人员检查手环 → 放行
```

**JWT_SECRET 是什么？**
- 是制作手环的模具
- 生成手环用模具 A，验证用模具 B → 对不上 ❌
- **服务器上的模具必须一致**

> ⚠️ **血泪教训**：上次 JWT_SECRET 不一致，登录一直失败！

### 3.8 什么是 CORS

**比喻**：小区门禁

```
浏览器：我要访问 https://pl.7025700.vip/api/login
   ↓
浏览器检查：这个请求是跨域的，需要服务器允许
   ↓
服务器响应头包含：
  Access-Control-Allow-Origin: https://pl.7025700.vip ✅
   ↓
浏览器：好，服务器同意了，放行

如果服务器没返回这个头：
   ↓
浏览器：服务器没同意，拦截 ❌
```

> ⚠️ **血泪教训**：上次 nginx 没配 CORS，浏览器一直拦截请求！

### 3.9 什么是 Nginx

**比喻**：门卫 + 调度员

```
互联网请求
   ↓
┌─────────────────┐
│     Nginx       │ ← 门卫
└─────────────────┘
   ↓         ↓
前端静态文件   后端 API
(/var/www/)   (localhost:3002)
```

**Nginx 负责**：
1. HTTPS 加密（SSL 证书）
2. 静态文件服务（HTML、CSS、JS）
3. API 请求转发（代理到后端）
4. CORS 头添加（门禁卡）

## 第四部分：版本管理（到底在管什么）

### 4.1 为什么需要版本管理

**问题场景**：
```
第 1 天：项目上线了
第 2 天：改了个 Bug
第 3 天：又加了个功能
第 4 天：改错了，想回到第 1 天...
   ↓
没有版本管理：回不去 😭
有版本管理：git checkout 第 1 天 ✅
```

### 4.2 Git 是什么

**比喻**：时间机器 + 保险箱

```
Git 仓库
├── 能回到过去（历史记录）
├── 能保存快照（提交）
├── 能贴标签（版本）
└── 能对比差异（diff）
```

### 4.3 版本管理流程

```
┌─────────────────────────────────────────┐
│ 1. 开发前                                │
│    git status → 确认干净                 │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 2. 开发功能                              │
│    改代码...                             │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 3. 本地构建测试                          │
│    pnpm run build                        │
│    确认没问题                            │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 4. 提交代码                              │
│    git add .                            │
│    git commit -m "描述本次修改"          │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 5. 部署到服务器                          │
│    ./deploy/sync-to-server.sh           │
│    先清旧文件 + 上传新文件                │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 6. 打版本标签                            │
│    git tag -a v1.2.0 -m "版本说明"       │
│    方便以后追溯                          │
└─────────────────────────────────────────┘
```

### 4.4 版本号命名

```
v1.2.3
│ │ │
│ │ └─ 修订号（Bug 修复，小改动）
│ └─── 次版本号（新功能，向后兼容）
└───── 主版本号（大改动，可能不兼容）
```

| 版本号 | 含义 | 示例 |
|--------|------|------|
| v1.0.0 | 首次上线 | 平台正式发布 |
| v1.0.1 | 修了个小 Bug | 修复登录按钮颜色 |
| v1.1.0 | 加了新功能 | 新增题库管理 |
| v2.0.0 | 大改版 | 架构重构 |

### 4.5 如何避免旧文件堆积

**问题根源**：
```
Vite 构建原理：
第 1 次构建 → index-ABC.js
第 2 次构建 → index-DEF.js
第 3 次构建 → index-GHI.js
   ↓
不删除旧的 → 服务器上堆积几十上百个文件！
```

**解决方案**：
```bash
# 部署脚本里已经包含
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"
```

**每次上传前先清空，确保只有当前版本**

## 第八部分：环境问题排查

### 8.1 常见问题速查

| 现象 | 可能原因 | 解决方法 |
|------|----------|----------|
| 页面显示旧内容 | 浏览器缓存 | Ctrl+Shift+R 强制刷新 |
| 登录返回 401 | JWT_SECRET 不一致 | 检查服务器.env 文件 |
| API 请求失败 | CORS 配置缺失 | 检查 nginx 配置 |
| 功能部分正常 | 前后端版本不匹配 | 重新运行同步脚本 |
| 服务无法访问 | 后端进程挂了 | pm2 restart |
| 端口被占用 | 另一个服务在用 | netstat -tlnp 查占用 |

### 8.2 排查步骤

```
1. 确认问题现象
   ↓
2. 检查服务器文件（SSH 登录查看）
   ↓
3. 检查服务状态（pm2 status）
   ↓
4. 检查日志（pm2 logs）
   ↓
5. 测试 API（curl 直接调用）
   ↓
6. 定位问题并修复
   ↓
7. 重新部署验证
```

## 第九部分：真实案例复盘 — "登录失效"风波

> 这是刚刚发生的真实故事，完整记录 AI 如何把一个好项目"搞乱"又如何"找回"。

### 9.1 事件背景

**时间**：2026-03-21

**起因**：用户要求把项目品牌名从"盼蕾"改为"若容"

**初始状态**：项目运行正常，用户访问 https://pl.7025700.vip

### 9.2 问题是如何发生的（一步步走偏）

```
第 1 步：改源代码 ✅
├─ 修改 LoginView.vue：盼蕾 → 若容
├─ 修改 index.html：盼蕾 → 若容
└─ 问题：没有重新构建 ❌
   └─ 以为：改了代码就生效
   └─ 实际：dist/ 还是旧文件

第 2 步：上传到服务器 ❌
├─ 运行部署脚本
├─ 问题：没有清空服务器旧文件
└─ 结果：新旧文件混在一起

第 3 步：用户反馈还是"盼蕾" ❌
├─ 原因：dist/ 没重新构建
└─ 浏览器读的是旧 index.html

第 4 步：重新构建上传 ✅
├─ 运行 pnpm run build:frontend
├─ 但：服务器上堆积了大量旧 JS 文件
└─ 问题：assets/ 目录有几十个旧版本

第 5 步：用户反馈登录失效 ❌
├─ 现象：输入账号密码 → 返回 401
├─ 错误提示："登录已失效，请重新登录"
└─ 问题升级：不只是名字问题

第 6 步：排查发现后端代码没同步 ❌
├─ 前端有 lookupByUsername 接口调用
└─ 后端没有这个方法 → 返回 404

第 7 步：修复后端，又发现 JWT 问题 ❌
├─ 服务器上有两个 .env 文件
├─ JWT_SECRET 不一致
└─ 生成 Token 用钥匙 A，验证用钥匙 B → 失败

第 8 步：修复 JWT，又发现 CORS 问题 ❌
├─ 浏览器请求被拦截
└─ nginx 没有配置 CORS 头

第 9 步：全部修复完成 ✅
└─ 登录正常，显示"若容"
```

### 9.4 AI 是如何被"带偏"的

**AI 的思维过程**：

```
用户：登录失效了

AI 第 1 轮思考：
├─ 假设 1：Token 过期了？
├─ 检查：JWT 有效期正常 ✅
└─ 结论：不是这个问题

AI 第 2 轮思考：
├─ 假设 2：后端 API 有问题？
├─ 测试：curl 调用登录 API
└─ 结果：返回 token ✅ — 说明后端正常

AI 第 3 轮思考：
├─ 矛盾：curl 正常，浏览器失败
├─ 假设 3：浏览器缓存？
├─ 建议：清除缓存/无痕模式
└─ 结果：还是失败 ❌

AI 第 4 轮思考：
├─ 关键线索：curl 能拿到 token，但浏览器说 401
├─ 洞察：同一个后端，为什么两种结果？
├─ 假设 4：CORS 问题！
├─ 检查：nginx 配置
└─ 发现：没有 Access-Control-Allow-Origin ❌

AI 第 5 轮思考：
├─ 验证：手动添加 CORS 头
├─ 测试：浏览器登录
└─ 结果：成功 ✅
```

**为什么会被带偏？**

| 轮次 | AI 怀疑的方向 | 为什么错 |
|------|--------------|---------|
| 第 1 轮 | Token 过期 | 只看了配置，没看实际验证 |
| 第 2 轮 | 后端 API 问题 | curl 测试成功，排除 |
| 第 3 轮 | 浏览器缓存 | 常见原因，但这次不是 |
| 第 4 轮 | CORS 问题 | 命中！ curl 和浏览器行为不同 |

**关键点**：
- AI 不是全知全能，是靠**排除法**一步步缩小范围
- 每次测试排除一个可能性
- 直到找到真正的根因

### 9.5 如何一步步找回新版本

**第 1 步：确认服务器文件是否正确**

```bash
# 查看服务器 index.html 标题
ssh root@120.77.144.5 "cat /var/www/panlei/index.html | grep title"

# 期望输出：<title>若容虚拟数据平台</title>
# 实际输出：<title>若容虚拟数据平台</title> ✅
```

**第 2 步：确认后端 API 是否正常**

```bash
# 测试 lookup-by-username 接口
curl 'https://pl.7025700.vip/api/auth/lookup-by-username?username=admin'

# 期望：返回机构信息
# 实际：Cannot GET /api/auth/lookup-by-username ❌

```

**第 3 步：同步后端代码**

```bash
# 上传后端源码
scp -r packages/backend/src root@120.77.144.5:/root/panlei/packages/backend/

# 服务器上重新构建
ssh root@120.77.144.5 "cd /root/panlei/packages/backend && pnpm run build"

# 重启服务
ssh root@120.77.144.5 "pm2 restart panlei-backend"
```

**第 4 步：检查 JWT 配置**

```bash
# 查看服务器上的 JWT_SECRET
ssh root@120.77.144.5 "cat /root/panlei/packages/backend/.env | grep JWT"

# 对比本地配置
cat packages/backend/.env | grep JWT

# 发现不一致 → 修改为相同值
```

**第 5 步：修复 CORS**

```bash
# 编辑 nginx 配置
ssh root@120.77.144.5 "nano /etc/nginx/sites-available/panlei"

# 添加：
location /api/ {
    proxy_set_header Origin $http_origin;

    if ($request_method = OPTIONS) {
        add_header Access-Control-Allow-Origin $http_origin;
        add_header Access-Control-Allow-Methods "GET, POST, OPTIONS";
        add_header Access-Control-Allow-Headers "Authorization, Content-Type";
        return 204;
    }
}

# 重启 nginx
ssh root@120.77.144.5 "nginx -s reload"
```

**第 6 步：验证登录**

```bash

# 输入 admin/admin123
# 结果：登录成功 ✅
```

### 9.6 如何清理旧文件

**问题**：服务器上有数十个旧 JS/CSS 文件

**原因**：Vite 构建原理

```
第 1 次构建：index-Ck6_hncp.js  (哈希值 hncp)
第 2 次构建：index-D7xQM2z.js   (哈希值 D7xQM2z)
第 3 次构建：index-Bv9Kp3L.js   (哈希值 Bv9Kp3L)
...
```

**为什么文件名变化？**
- Vite 自动给文件加哈希值
- 内容变了 → 哈希值变 → 文件名变
- 好处：浏览器缓存不会冲突
- 坏处：不删除旧文件会堆积

**解决方案**：部署脚本已修复

```bash
# 上传前先清空
ssh root@120.77.144.5 "rm -rf /var/www/panlei/assets/*"

# 再上传新文件
scp -r packages/frontend/dist/* root@120.77.144.5:/var/www/panlei/
```

### 9.7 经验教训总结

| 教训 | 应该怎么做 | 已落实的措施 |
|------|-----------|-------------|
| 改代码必须重新构建 | 改完立即 `pnpm run build` | 部署脚本自动构建 |
| 上传前清空旧文件 | `rm -rf assets/*` | sync-to-server.sh 已包含 |
| 前后端要同步更新 | 一起修改 + 一起部署 | 部署脚本同时上传前后端 |
| 配置只能有一份 | 统一 JWT_SECRET | 文档明确说明 |
| CORS 必须配置 | nginx 转发 Origin 头 | 配置已固化 |
| 版本要打标签 | `git tag -a v1.x.x` | Git 仓库已建立 |

## 第十一部分：常见疑问解答

### 11.1 为什么 AI 会"忘记"构建？

**用户的疑问**：
> "你不构建根本就运行不了，我也不可能看到结果，但每次修改后我都能看到结果"

**真相揭秘**：

```
┌─────────────────────────────────────────────────────────┐
│ 本地开发模式（pnpm dev）                                │
├─────────────────────────────────────────────────────────┤
│ 你改代码 → Vite 自动实时编译 → 浏览器自动刷新           │
│ 所以：你每次修改都能立即看到结果 ✅                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 生产部署模式（pnpm run build）                          │
├─────────────────────────────────────────────────────────┤
│ 你改代码 → 必须手动运行 build → 生成新的 dist/          │
│ 否则：上传的还是旧的 dist/ ❌                           │
└─────────────────────────────────────────────────────────┘
```

**为什么会有这个矛盾？**

| 场景 | 构建方式 | 文件名 | 是否需要手动执行 |
|------|---------|--------|-----------------|
| 本地开发（pnpm dev） | Vite 自动实时构建 | 不变 | 不需要 |
| 生产部署（pnpm run build） | 手动执行 | 带哈希 | 需要 |

**AI 犯错的过程**：
```
1. AI 用 pnpm dev 测试 → 自动更新 → 看到效果 ✅
2. AI 以为 dist/ 已经是最新的 → 其实不是 ❌
3. 上传旧 dist/ 到服务器 → 显示旧内容 ❌
```

**根本原因**：
- `pnpm dev` 模式的 dist/ 和 `pnpm run build` 的 dist/ 是**两回事**
- 开发模式：代码不压缩，方便调试
- 生产模式：代码压缩合并，优化性能

---

### 11.2 构建（Build）到底是什么？

**比喻**：装修房子

```
源代码（src/）              构建（Build）            构建产物（dist/）
┌─────────────────┐        ┌─────────┐        ┌─────────────────┐
│ 设计图纸         │        │         │        │ 装修好的房子     │
│ 木材 + 油漆      │  ──→   │  施工队 │  ──→   │ 可以入住         │
│ 原始材料         │        │  机器   │        │ 给用户使用       │
└─────────────────┘        └─────────┘        └─────────────────┘
   给人看的                  加工处理               给客人住的
```

**具体变化**：

你写的代码（源代码）：
```vue
<!-- LoginView.vue -->
<template>
  <div class="login-page">
    <h1>若容虚拟数据平台</h1>
  </div>
</template>

<script lang="ts">
// TypeScript 代码，浏览器看不懂
</script>
```

构建后（dist/）：
```html
<!-- index.html -->
<!DOCTYPE html>
<html>
<head>
  <title>若容虚拟数据平台</title>
  <script src="/assets/index-Ck6_hncp.js"></script>
  <!-- 机器生成的文件，人看不懂但浏览器能跑 -->
</html>
```

**构建过程做了 4 件事**：

| 步骤 | 做了什么 | 比喻 |
|------|---------|------|
| 1. 翻译 | TypeScript → JavaScript | 中文翻译成英文 |
| 2. 合并 | 100 个小文件 → 3 个大文件 | 零散食材打包成套餐 |
| 3. 压缩 | 去掉空格/注释，变量名缩短 | 真空压缩，减小体积 |
| 4. 加哈希 | 文件名加上内容指纹 | 给产品贴防伪标签 |

### 11.3 为什么改了代码必须重新构建？

**错误做法**：
```
改了 LoginView.vue（源代码）
   ↓
直接上传到服务器（没构建）
   ↓
服务器还是旧的 dist/
   ↓
用户看到的还是旧页面 ❌
```

**正确做法**：
```
改了 LoginView.vue（源代码）
   ↓
运行 pnpm run build:frontend（重新构建）
   ↓
生成新的 dist/（包含最新修改）
   ↓
上传新的 dist/ 到服务器
   ↓
用户看到新页面 ✅
```
