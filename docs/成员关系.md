# 超级管理员 - 机构成员管理增强设计方案

> 文档版本：v1.0

> 状态：待确认

## 一、核心需求分析

| 需求点 | 说明 |
|--------|------|
| 编辑成员时可切换机构 | 编辑弹窗中增加「所属机构」下拉选择器 |
| 编辑成员时可修改角色 | 角色下拉框包含所有可选角色 |
| 删除成员时软删除 | 标记 `isDeleted`，数据保留可找回 |
| 删除警告提示 | 明确告知该用户所有资料将被删除（考试记录/成绩/评分等） |
| 软删除用户可恢复 | 提供「已删除用户」列表和恢复功能 |
| 分级管理权限 | 各级只能管理下一级，不能管理同级和上级 |

## 二、角色层级体系设计

```
┌─────────────────────────────────────────────────────────────┐
│ 角色层级（从上到下，管理权限逐级向下）                        │
├─────────────────────────────────────────────────────────────┤
│  SUPER_ADMIN（超级管理员）- 平台级，管理所有机构               │
├─────────────────────────────────────────────────────────────┤
│  TENANT_ADMIN（机构管理员）- 机构级，管理本机构所有用户         │
├─────────────────────────────────────────────────────────────┤
│  CLASS_ADMIN（班级管理员）- 班级级，管理本班所有用户           │
│  【新增】原 CLASS 角色更名为 CLASS_ADMIN                       │
├─────────────────────────────────────────────────────────────┤
│  TEACHER（教师）- 可管理学生                                  │
├─────────────────────────────────────────────────────────────┤
│  STUDENT（学生）- 无管理权限                                  │
└─────────────────────────────────────────────────────────────┘
```

## 三、分级管理权限矩阵

| 当前用户角色 | 可查看/管理下一级 | 不可查看/管理同级 | 不可查看/管理上级 |
|-------------|------------------|------------------|------------------|
| TENANT_ADMIN | CLASS_ADMIN, TEACHER, STUDENT | 其他 TENANT_ADMIN | SUPER_ADMIN |
| CLASS_ADMIN | TEACHER, STUDENT | 其他 CLASS_ADMIN | TENANT_ADMIN, SUPER_ADMIN |
| TEACHER | STUDENT | 其他 TEACHER | CLASS_ADMIN, TENANT_ADMIN, SUPER_ADMIN |
| STUDENT | 无 | - | 所有角色 |

## 四、数据库设计变更

### 4.1 User 表新增字段

```prisma
model User {
  // ... 现有字段保留 ...
  isDeleted  Boolean  @default(false)  // 【新增】软删除标记
  deletedAt  DateTime?                 // 【新增】删除时间
  deletedBy  String?  @db.VarChar(191) // 【新增】删除人 ID

  // 索引
  @@index([tenantId, isDeleted])
  @@index([deletedAt])
}
```

### 4.2 角色枚举更新

```prisma
enum UserRole {
  SUPER_ADMIN      // 超级管理员
  TENANT_ADMIN     // 机构管理员
  CLASS_ADMIN      // 【更名】原 CLASS
  TEACHER          // 教师
  STUDENT          // 学生
}
```

## 五、后端接口设计

### 5.1 机构成员管理 - 增强接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/admin/tenants/:tenantId/users` | 机构成员列表（支持筛选是否已删除） |
| PATCH | `/admin/tenants/:tenantId/users/:userId` | 编辑成员（可改机构/角色） |
| DELETE | `/admin/tenants/:tenantId/users/:userId` | 软删除成员 |
| POST | `/admin/tenants/:tenantId/users/:userId/restore` | 恢复已删除成员 |
| GET | `/admin/tenants/:tenantId/users/deleted` | 已删除成员列表 |
| POST | `/admin/tenants/:tenantId/users/:userId/permanent-delete` | 永久删除（超管专用） |

### 5.2 分级管理接口（各角色通用）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/users/subordinates` | 获取当前用户可管理的下级列表 |
| POST | `/users/subordinates` | 新增下级用户 |
| PATCH | `/users/subordinates/:userId` | 编辑下级用户 |
| DELETE | `/users/subordinates/:userId` | 软删除下级用户 |
| GET | `/users/subordinates/deleted` | 已删除的下级用户列表 |
| POST | `/users/subordinates/:userId/restore` | 恢复下级用户 |

## 七、关键业务逻辑

### 7.1 软删除逻辑

```typescript
async softDeleteUser(userId: string, operatorId: string) {
  // 1. 标记用户为已删除
  await this.prisma.user.update({
    where: { id: userId },
    data: {
      isDeleted: true,
      deletedAt: new Date(),
      deletedBy: operatorId,
    },
  });

  // 2. 标记关联资料为已删除（根据选项）
  // - ExamParticipant
  // - ExamAnswer
  // - Score
  // - ScoreRecord
}
```

### 7.2 恢复用户逻辑

```typescript
async restoreUser(userId: string, operatorId: string) {
  // 1. 恢复用户状态
  await this.prisma.user.update({
    where: { id: userId },
    data: {
      isDeleted: false,
      deletedAt: null,
      deletedBy: null,
    },
  });

  // 2. 恢复关联资料
  // ...
}
```

### 7.3 分级权限校验

```typescript
async canManageUser(managerRole: UserRole, targetRole: UserRole): Promise<boolean> {
  const roleHierarchy: Record<UserRole, UserRole[]> = {
    SUPER_ADMIN: [TENANT_ADMIN, CLASS_ADMIN, TEACHER, STUDENT],
    TENANT_ADMIN: [CLASS_ADMIN, TEACHER, STUDENT],
    CLASS_ADMIN: [TEACHER, STUDENT],
    TEACHER: [STUDENT],
    STUDENT: [],
  };

  return roleHierarchy[managerRole]?.includes(targetRole) ?? false;
}
```

## 九、已确认问题

| 问题 | 确认方案 |
|------|----------|
| 教师管理学生范围 | 教师只能管理所教班级学生，不能管理本机构所有学生 |
| 角色切换限制 | 机构管理员不能通过编辑用户的方式直接提升，需在专门菜单中设定 |
| 软删除后登录 | 已删除用户禁止登录 |
| 软删除后数据可见性 | 考试成绩等数据对原用户隐藏 |
| 永久删除权限 | 仅 SUPER_ADMIN 可永久删除 |
| 角色更名影响 | CLASS→CLASS_ADMIN，需要更新所有引用 |

## 十一、学生分配流程说明

### 11.1 学生的来源

```
┌─────────────────────────────────────────────────────────────┐
│  学生创建流程                                                │
├─────────────────────────────────────────────────────────────┤
│  1. 机构管理员/班级管理员 → 用户管理 → 新增学生              │
│  2. 填写信息（用户名、姓名、所属机构/班级）                   │
│  3. 系统创建学生账号                                         │
│  4. 学生自动归属到指定班级                                   │
└─────────────────────────────────────────────────────────────┘
```

### 11.2 教师与学生的关联

```
┌─────────────────────────────────────────────────────────────┐
│  教师管理学生流程                                            │
├─────────────────────────────────────────────────────────────┤
│  方式一：通过班级管理                                        │
│    机构管理员 → 班级管理 → 编辑班级 → 设置授课教师           │
│    → 该教师自动获得该班级学生的管理权限                       │
│                                                              │
│  方式二：通过用户分配                                        │
│    机构管理员 → 用户管理 → 编辑教师 → 设置所教班级           │
│    → 该教师自动获得该班级学生的管理权限                       │
└─────────────────────────────────────────────────────────────┘
```

### 11.3 教师对学生的管理权限边界

| 操作类型 | 教师是否可操作 | 说明 |
|----------|:--------------:|------|
| 创建学生 | ❌ | 只能由机构管理员/班级管理员创建 |
