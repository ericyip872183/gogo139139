> **一句话总结**：本帮助你从混乱中建立起清晰的版本管理体系，确保每次部署都可追溯、可回滚、可验证。

## 第一部分：我们经历了什么（问题回顾）

### 1.1 事件时间线

```
3 月 18 日  项目首次部署成功
    ↓
3 月 21 日  你要求改"盼蕾"为"若容"
    ↓
我改了源代码（LoginView.vue、index.html 等）✅
    ↓
但我没有重新构建就上传了 ❌
    ↓
你看到服务器还是"盼蕾"
    ↓
我重新构建并上传，但没删服务器旧文件 ❌
    ↓
服务器堆积了多个版本的 JS 文件
    ↓
你反馈登录失效
    ↓
我发现后端代码没同步 ❌
    ↓
同步后端，但 JWT_SECRET 不一致 ❌
    ↓
修复 JWT，但 nginx CORS 配置缺失 ❌
    ↓
最终修复完成 ✅
```

### 1.2 问题本质

| 表面现象 | 根本原因 |
|----------|----------|
| 服务器显示旧内容 | 改了源代码但没重新构建 |
| 登录失效 | 后端代码没同步 + JWT 密钥不一致 |
| 浏览器报错 | nginx 缺少 CORS 配置 |
| 文件混乱 | 每次上传不删除旧文件 |

## 第二部分：核心概念解释（说人话）

### 2.1 源代码 vs 构建产物

```
源代码（src/）          构建（build）         构建产物（dist/）
┌─────────────┐        ┌─────────┐        ┌─────────────┐
│ LoginView   │        │         │        │ index-ABC.js│ ← 浏览器读的
│ index.html  │  ──→   │  Vite   │  ──→   │ index-XYZ.css│
│ main.ts     │        │         │        │ ...         │
└─────────────┘        └─────────┘        └─────────────┘
   给人看的              机器处理             给浏览器看的
```

**关键理解**：
- 你改的是 `src/` 下的源代码
- 浏览器用的是 `dist/` 下的构建产物
- **改完源代码必须运行 `pnpm run build` 才能生效**

类比：源代码是食材，dist 是做好的菜。你换了食材，必须重新炒一遍，客人才能吃到新菜。

### 2.2 前端 vs 后端

```
用户浏览器
    ↓
前端（Vue）：负责显示页面、收集用户输入
    ↓ HTTP 请求（/api/...）
后端（NestJS）：负责处理业务逻辑、读写数据库
    ↓
数据库（MySQL）：存储用户、题目、成绩等数据
```

**关键理解**：
- 前端是"壳"，后端是"核"
- 前端改了要重新构建上传
- 后端改了要重新构建并重启服务
- **前后端必须同时更新，否则会脱节**

### 2.3 Token 与 JWT

```
登录成功
    ↓
后端生成 Token（加密的身份证明）
    ↓
前端把 Token 存到 localStorage
    ↓
每次请求都带上 Token
    ↓
后端验证 Token 有效性
    ↓
验证通过 → 返回数据
验证失败 → 返回 401"登录已失效"
```

**JWT_SECRET 是什么？**
- 是生成和验证 Token 的"钥匙"
- 生成 Token 用钥匙 A，验证用钥匙 B → 验证失败
- **服务器上的 JWT_SECRET 必须始终一致**

### 2.4 CORS（跨域资源共享）

```
浏览器：我要访问 https://pl.7025700.vip/api/login
    ↓
浏览器检查：这个请求是跨域的，需要服务器允许
    ↓
服务器响应头包含：
  Access-Control-Allow-Origin: https://pl.7025700.vip
    ↓
浏览器：好，服务器同意了，放行
    ↓
请求成功

如果服务器没返回这个头：
    ↓
浏览器：服务器没同意，拦截请求
    ↓
前端收不到响应，或者收到 401
```

**类比**：
- CORS 像是小区的门禁系统
- 没有门禁卡（CORS 头），保安（浏览器）不让你进

### 2.5 Nginx 的作用

```
互联网请求
    ↓
┌─────────────────┐
│     Nginx       │ ← 门卫 + 调度员
└─────────────────┘
    ↓         ↓
前端静态文件   后端 API
(/var/www/)   (localhost:3002)
```

**Nginx 负责**：
1. HTTPS 加密（SSL 证书）
2. 静态文件服务（HTML、CSS、JS）
3. API 请求转发（代理到后端）
4. CORS 头添加

## 第三部分：版本管理体系

### 3.1 为什么需要 Git

**没有 Git 时**：
- 不知道改了什么
- 改错了无法回滚
- 本地和服务器无法对比

**有 Git 后**：
- 每次修改都有记录
- 随时可以回到历史版本
- 可以生成版本标签

### 3.2 Git 基础概念

```
工作区 → 暂存区 → 本地仓库 → 远程仓库
        (git add) (git commit) (git push)
```

| 概念 | 说明 | 类比 |
|------|------|------|
| 工作区 | 你正在编辑的文件 | 书桌上的文件 |
| 暂存区 | 准备提交的文件 | 待归档的文件盒 |
| 提交 | 保存版本快照 | 存入档案室 |
| 分支 | 独立的开发线 | 平行宇宙 |
| 标签 | 给版本起名字 | 档案盒上的标签 |

### 3.3 本地 Git vs GitHub

| 本地 Git 仓库 | GitHub |
|--------------|--------|
| 在你的电脑里 | 在互联网服务器上 |
| 不需要网络 | 需要网络 |
| 完全私有 | 可以选择公开或私有 |
| 免费 | 基础功能免费 |
| 只有你能访问 | 可以协作、分享 |

**类比**：
- 本地 Git = 你家中的保险箱
- GitHub = 银行的保险库

**建议**：
1. **必须**建立本地 Git 仓库（基础版本控制）
2. GitHub 可选（如果需要备份或协作）

### 3.4 版本命名规范

采用语义化版本号：`v<主版本>.<次版本>.<修订版本>`

| 版本号 | 说明 | 何时使用 |
|--------|------|----------|
| v1.0.0 | 初始版本 | 首次上线 |
| v1.0.1 | Bug 修复 | 修复登录问题 |
| v1.1.0 | 新功能 | 新增题库模块 |
| v2.0.0 | 重大更新 | 架构重构 |

## 第四部分：标准化部署流程

### 4.1 部署前准备

```bash
# 1. 确认本地代码已提交
cd c:\项目\panlei
git status
# 应该显示 "working tree clean"

# 2. 查看当前版本
git log -1 --oneline
# 例如：a1b2c3d 修复登录 API

# 3. 打版本标签
git tag -a v1.1.1 -m "修复登录 API + CORS + JWT 配置"
```

### 4.2 一键部署脚本

`deploy/sync-to-server.sh` 已包含完整流程：

```bash
#!/bin/bash
# 使用方法：./deploy/sync-to-server.sh

# 1. 构建前端
pnpm run build:frontend

# 2. 验证构建
检查 dist/index.html 是否存在且标题正确

# 3. 清空服务器旧文件
ssh "rm -rf /var/www/panlei/assets/*"

# 4. 上传前端
scp dist/* 服务器:/var/www/panlei/

# 5. 上传后端代码
scp src/* 服务器:/root/panlei/packages/backend/

# 6. 服务器上构建后端
ssh "pnpm install && pnpm prisma generate && pnpm run build"

# 7. 重启后端服务
ssh "pm2 restart panlei-backend"

# 8. 验证
检查服务状态和 API 响应
```

### 4.3 部署后验证

```bash
# 1. 检查前端版本
curl https://pl.7025700.vip | grep title
# 应显示：若容虚拟数据平台

# 2. 检查后端 API
curl https://pl.7025700.vip/api/auth/lookup-by-username?username=admin

# 3. 测试登录
用 admin/admin123 登录，确认能进入系统

# 4. 清除浏览器缓存
Ctrl+Shift+R 强制刷新
```

## 第五部分：端口与进程管理

### 5.1 服务器端口分配（固定）

| 服务 | 端口 | 说明 |
|------|------|------|
| 若容后端 | 3002 | API 服务 |
| PulseChat | 3000 | 另一个项目 |
| MySQL | 3306 | 数据库 |
| Redis | 6379 | 缓存 |
| Nginx | 80 | HTTP |
| Nginx | 443 | HTTPS |
| Nginx | 8090 | 测试访问 |

### 5.2 PM2 进程管理

```bash
# 查看所有进程
pm2 status

# 重启若容后端
pm2 restart panlei-backend

# 查看日志
pm2 logs panlei-backend --lines 50

# 开机自启（只需配置一次）
pm2 startup
pm2 save
```

### 5.3 避免端口冲突

1. **每个项目固定端口**：若容永远用 3002
2. **nginx 按域名分发**：不同域名到不同端口
3. **启动前检查端口占用**：`netstat -tlnp | grep 3002`

## 第六部分：确保版本一致性

### 6.1 版本号嵌入代码

在 `package.json` 中维护版本号：
```json
{
  "name": "frontend",
  "version": "1.1.1"
}
```

在页面底部显示：
```vue
<div class="app-footer">
  若容平台 v{{ version }}
</div>
```

### 6.2 部署验证脚本

创建 `deploy/verify-deployment.sh`：
```bash
#!/bin/bash

echo "=== 验证部署 ==="

# 前端版本
LOCAL_HTML=$(cat packages/frontend/dist/index.html | grep title)
SERVER_HTML=$(ssh root@120.77.144.5 "cat /var/www/panlei/index.html | grep title")

echo "本地：$LOCAL_HTML"
echo "服务器：$SERVER_HTML"

if [ "$LOCAL_HTML" = "$SERVER_HTML" ]; then
  echo "✅ 前端版本一致"
else
  echo "❌ 前端版本不一致"
fi

# 后端版本
LOCAL_HASH=$(git log -1 --oneline)
SERVER_HASH=$(ssh root@120.77.144.5 "cd /root/panlei && git log -1 --oneline")

echo "本地 Git: $LOCAL_HASH"
echo "服务器 Git: $SERVER_HASH"

# 服务状态
ssh root@120.77.144.5 "pm2 status panlei-backend"
```

### 6.3 部署检查清单

- [ ] 访问 https://pl.7025700.vip 验证页面

## 第七部分：故障排查指南

### 7.1 常见问题速查

| 现象 | 可能原因 | 解决方法 |
|------|----------|----------|
| 页面显示旧内容 | 浏览器缓存 | Ctrl+Shift+R 强制刷新 |
| 登录返回 401 | JWT_SECRET 不一致 | 检查服务器.env 文件 |
| API 请求失败 | CORS 配置缺失 | 检查 nginx 配置 |
| 功能部分正常 | 前后端版本不匹配 | 重新运行同步脚本 |
| 服务无法访问 | 后端进程挂了 | pm2 restart |

### 7.2 排查步骤

```
1. 确认问题现象
   ↓
2. 检查服务器文件（SSH 登录查看）
   ↓
3. 检查服务状态（pm2 status）
   ↓
4. 检查日志（pm2 logs）
   ↓
5. 测试 API（curl 直接调用）
   ↓
6. 定位问题并修复
   ↓
7. 重新部署验证
```

## 第九部分：总结与行动清单

### 9.1 核心思想

1. **源代码≠构建产物**：改了代码必须重新构建
2. **前后端要同步**：单独更新一个会脱节
3. **删除旧文件**：上传新文件前先清空
4. **配置要一致**：JWT_SECRET 等关键配置只能有一份
5. **版本要标签**：每次部署打标签，方便追溯

### 9.2 立即执行

```bash
# 1. 初始化 Git 仓库
cd c:\项目\panlei
git init
git add .
git commit -m "Initial commit: 若容平台 v1.1.1"

# 2. 打当前版本标签
git tag -a v1.1.1 -m "修复登录 API + CORS + JWT 配置"

# 3. 测试部署脚本
./deploy/sync-to-server.sh

# 4. 验证部署
curl https://pl.7025700.vip | grep title
```

### 9.3 长期维护

| 频率 | 任务 |
|------|------|
| 每次修改代码后 | git commit + 描述 |
| 每次部署前 | git tag 打版本 |
| 每月 | 检查服务器磁盘空间 |
| 每季度 | 更新依赖包版本 |
