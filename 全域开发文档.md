# 全局地图 — 中医/医学教学在线考试学习系统

---

## 1. 产品一句话定位

中医/医学 SaaS 一体化教学考试平台 = 考试母本系统（核心功能全模块共用）+ 可插拔专业教学模块（独立界面）+ 硬件实操对接 + 多维度统一评分，面向中医/医学院校，覆盖理论教学、在线考试、实训操作、技能考核全场景。

### 核心依赖链

- **所有模块** → 登录认证 + 用户组织 + SaaS授权（必须先有母本底座）
- **所有含考核的模块** → 题库 + 组卷 + 阅卷成绩（共用母本考试能力）
- **所有含评分的模块** → 评分体系（共用母本评分规则）
- **硬件类模块（经络/四诊/针刺/刮痧/推拿/腧穴解剖）** → 硬件通信标准模块
- **体质辨识、耳穴教学** → 不依赖硬件通信层，可独立开发

| 分期 | 内容 | 理由 |

| **一期（已完成）** | 母本核心 8 模块：登录→用户→题库→组卷→考试→阅卷→评分→SaaS授权 + 硬件通信标准模块 | 这是整个平台的底座，所有拓展模块都依赖它，必须先行。已完成 Sprint 1-9 |

| **三期** | 经络采集 + 四诊采集 + 针刺手法 + 刮痧手法 + 推拿手法 + 经络腧穴解剖 + 外部软件对接 | 全部依赖硬件设备，需要硬件厂商配合联调，开发周期长、不确定性高，放最后 |

## 6. 最大的技术风险点

| 风险 | 等级 | 说明 |
|------|------|------|
| **硬件协议对接不确定性** | 🔴 高 | 6个模块依赖不同厂商的硬件设备（传感器/力反馈臂/VR头显），协议文档质量、联调周期完全不可控，是最大的进度风险 |
| **Web Serial / Web Bluetooth 浏览器兼容性** | 🔴 高 | 这两个 API 仅 Chromium 内核支持，Safari/Firefox 不支持，且必须 HTTPS 环境，部署时需强制要求 Chrome 89+ |
| **VR 模块（刮痧/针刺）的 Web 端可行性** | 🟡 中 | 策划书要求力反馈机械臂 + 6DoF VR 头显，这通常是原生应用场景，纯 Web 端实现存疑，可能需要本地客户端或 WebXR 方案 |
| **离线评分 + 断网续答的数据一致性** | 🟡 中 | Service Worker 缓存 + 网络恢复同步，需要处理冲突合并、幂等提交、数据版本控制，实现复杂度高 |
| **多租户数据隔离** | 🟡 中 | 所有查询必须带 tenantId，一旦遗漏就是数据泄露，随着模块增多，遗漏风险线性增长 |
| **实时双向数据同步（母本↔模块）** | 🟡 中 | 策划书反复强调"实时同步、无延迟"，如果模块是同一个数据库则天然满足，但如果未来模块独立部署则需要消息队列/事件总线 |

## 7. 技术选型与架构（基于现有代码库）

> 以下内容基于项目已有的 package.json、Dockerfile、docker-compose.yml、nginx.conf 等实际配置整理，不是重新选型，而是对现有选择的梳理和补充说明。

### 7.1 前端框架

| 项目 | 选型 | 版本 |
|------|------|------|
| 框架 | Vue 3 | ^3.4.0 |
| 语言 | TypeScript | ^5.3.3 |
| 构建 | Vite | ^5.0.11 |
| UI 库 | Element Plus | ^2.5.0 |
| 状态管理 | Pinia | ^2.1.7 |
| 路由 | Vue Router 4 | ^4.2.5 |
| HTTP | Axios | ^1.6.5 |
| 图表 | ECharts + vue-echarts | ^5.4.3 |
| 组件自动导入 | unplugin-auto-import + unplugin-vue-components | - |

为什么选 Vue 3 + Element Plus：
- 项目面向国内中医/医学院校，Vue 在国内生态最成熟，Element Plus 是 Vue 3 最主流的后台 UI 库
- Composition API + TypeScript 严格模式保证代码质量
- Vite 开发体验快，HMR 秒级刷新
- unplugin 自动导入减少样板代码，Element Plus 组件按需加载

### 7.2 后端框架

| 项目 | 选型 | 版本 |
|------|------|------|
| 框架 | NestJS | ^10.3.0 |
| ORM | Prisma | ^5.8.0 |
| 认证 | JWT + Passport | ^10.2.0 |
| 密码 | bcrypt | ^5.1.1 |
| 校验 | class-validator + class-transformer | ^0.14.1 |
| 实时通信 | Socket.io（NestJS 内置网关） | - |
| 邮件 | nodemailer | ^8.0.3 |

为什么选 NestJS + Prisma：
- NestJS 模块化架构天然适合 SaaS 多模块系统，每个业务模块独立 Module，解耦清晰
- 内置依赖注入、Guard、Interceptor、Pipe，多租户中间件和权限守卫开箱即用
- Prisma 类型安全的 ORM，schema 即文档，`prisma db push` 一键同步数据库
- 前后端统一 TypeScript，共享 `packages/shared` 中的类型定义和常量

### 7.3 数据库

| 项目 | 选型 | 版本 |
|------|------|------|
| 主数据库 | MySQL | 8.0 |
| 连接方式 | Prisma Client | ^5.8.0 |

为什么选 MySQL 8：
- 国内院校 IT 运维团队最熟悉的数据库，部署维护门槛低
- 8.0 支持 JSON 字段（评分记录 itemScores 用到）、窗口函数（排名计算）、CTE（组织树查询）
- Prisma 对 MySQL 支持成熟稳定

### 7.4 缓存

| 项目 | 选型 | 版本 |
|------|------|------|
| 缓存 | Redis | 7-alpine |
| 客户端 | ioredis | ^5.10.0 |

当前用途：
- JWT Token 黑名单（登出/踢人）
- 考试会话缓存（防止重复进入、记录答题进度）
- 验证码/短信码临时存储

未来扩展用途（二期）：
- Socket.io 适配器（多实例部署时共享 WebSocket 状态）
- 热点数据缓存（题库分类树、模块列表等低频变更数据）
- 分布式锁（防止并发交卷/重复评分）

### 7.5 文件存储方案

| 阶段 | 方案 | 说明 |
|------|------|------|
| 开发阶段（当前） | 本地磁盘 | 文件存 `uploads/` 目录，够用 |
| 生产部署 | 阿里云 OSS | .env 已预留 OSS_REGION / OSS_ACCESS_KEY_ID / OSS_BUCKET 配置项 |

存储内容：用户头像、题目图片、评分表附件、导入导出的 Excel/Word、二期的多媒体教学资料（视频/文档）

接入方式：后端封装统一的 `FileService`，开发环境写本地，生产环境写 OSS，通过环境变量切换，上层业务代码无感知。

### 7.6 本地开发环境

```yaml

    image: redis:7-alpine
```

本地开发启动流程：

```bash
# 1. 启动 MySQL + Redis
docker compose up -d

# 2. 安装依赖
pnpm install

# 3. 同步数据库
pnpm db:generate && pnpm --filter backend prisma db push

# 4. 前后端同时启动
pnpm dev
# → 后端: http://localhost:3000/api
# → 前端: http://localhost:5173 (自动代理 /api → 3000)
```

项目采用全容器化部署（已有完整配置）：

```

│  │  Nginx   │→│  Backend  │                 │
```

关键配置（已实现）：
- 前端：多阶段 Dockerfile，builder 阶段 `pnpm build`，runner 阶段用 nginx:alpine 托管静态文件

### 7.8 技术难点提前预警

| # | 难点 | 影响范围 | 建议应对 |
|---|------|---------|---------|
| 1 | **硬件协议对接** | 三期 6 个硬件模块 | 硬件通信标准模块（Sprint 9）已完成抽象层，但真实设备联调需要厂商提供协议文档，建议尽早拿到样机 |
| 2 | **Web Serial/Bluetooth 浏览器限制** | 所有硬件模块 | 仅 Chromium 内核支持，必须 HTTPS，部署时需强制 Chrome 89+，Safari/Firefox 用户无法使用硬件功能 |
| 3 | **VR 模块的 Web 端实现** | 刮痧/针刺模块 | 力反馈机械臂 + 6DoF VR 头显通常是原生应用场景，纯浏览器方案需评估 WebXR API 可行性，可能需要本地客户端桥接 |
| 4 | **离线评分 + 断网续答** | 考试/评分模块 | 需要 Service Worker 缓存策略 + IndexedDB 本地存储 + 网络恢复后的冲突合并逻辑，建议二期专项攻关 |
| 5 | **多租户隔离一致性** | 全系统 | 当前通过 TenantGuard 中间件注入 tenantId，随着模块增多需要持续 review 确保无遗漏，建议加集成测试覆盖 |
| 6 | **Socket.io 多实例** | PBL 实时讨论（二期） | 单实例够用，但如果未来水平扩展需要 Redis Adapter 共享 WebSocket 状态，架构上已预留 Redis |
| 7 | **大文件上传（视频/3D模型）** | 资料管理/3D 模块 | 阿里云 OSS 分片上传 + 断点续传，前端需要实现上传进度条和失败重试 |
| 8 | **考试防作弊** | 在线考试模块 | 当前已实现 Page Visibility API 防切屏，二期如需更强防作弊（摄像头监控、人脸识别）复杂度会大幅上升 |

## 8. 数据库设计

### 8.1 所有核心表（一句话说明用途）

| # | 表名 | 用途 |
|---|------|------|
| 1 | `tenants` | 机构（租户），SaaS 多租户隔离的根节点 |
| 2 | `users` | 所有用户（超管/教师/学生），通过 tenantId 归属机构 |
| 3 | `organizations` | 4级树形组织架构（学校→院系→专业→班级），支持自定义层级名称 |
| 4 | `user_orgs` | 用户与组织的多对多关联，一个用户可属于多个组织 |
| 5 | `question_categories` | 题目分类树，支持模块专属子分类（moduleCode 区分） |
| 6 | `questions` | 题目主体（单选/多选/判断/填空），含标准答案和解析 |
| 7 | `question_options` | 题目选项（A/B/C/D），标记正确答案 |
| 8 | `papers` | 试卷，由题目组合而成，设定总分和时长 |
| 9 | `paper_questions` | 试卷与题目的关联，可覆盖默认分值、控制顺序 |
| 10 | `exams` | 考试，关联试卷，设定开始/结束时间和考试规则 |
| 11 | `exam_participants` | 考试指定考生，记录交卷状态和切屏次数 |
| 12 | `exam_answers` | 学生答题记录，每题一条，支持自动判分 |
| 13 | `scores` | 考试成绩汇总，含总分/正确率/排名 |
| 14 | `score_tables` | 评分表定义（加分制/减分制），用于技能考核 |
| 15 | `score_items` | 评分表中的评分项，每项有分值和排序 |
| 16 | `score_records` | 考官对学生的打分记录，支持离线同步标记 |
| 17 | `modules` | 平台级模块定义（体质辨识/耳穴/针刺等），无 tenantId |
| 18 | `tenant_modules` | 机构模块授权，控制哪个机构购买了哪个模块及有效期 |

### 8.2 表与表之间的关系

```
Tenant (机构) ─── 多租户隔离根节点，以下所有表均通过 tenantId 挂靠
  │
  ├─→ User (用户) [tenantId]
  │     │
  │     ├─→ UserOrg ←─→ Organization    // 用户-组织 多对多
  │     ├─→ ExamParticipant              // 用户参加的考试
  │     ├─→ ExamAnswer                   // 用户的答题记录
  │     ├─→ Score                        // 用户的成绩
  │     ├─→ ScoreRecord (as judge)       // 作为考官的打分记录
  │     └─→ ScoreRecord (as target)      // 作为被评人的打分记录
  │
  ├─→ Organization (组织) [tenantId, parentId→self]  // 自引用树形结构
  │
  ├─→ QuestionCategory [tenantId, parentId→self]     // 自引用分类树
  │     └─→ Question [categoryId]
  │           └─→ QuestionOption [questionId]         // 级联删除
  │
  ├─→ Paper (试卷) [tenantId]
  │     └─→ PaperQuestion ←─→ Question               // 试卷-题目 多对多
  │
  ├─→ Exam (考试) [tenantId, paperId→Paper]
  │     ├─→ ExamParticipant ←─→ User                 // 考试-考生 多对多
  │     ├─→ ExamAnswer [examId, userId, questionId]   // 三元唯一
  │     └─→ Score [examId, userId]                    // 二元唯一
  │
  ├─→ ScoreTable (评分表) [tenantId]
  │     ├─→ ScoreItem [tableId]                       // 级联删除
  │     └─→ ScoreRecord [tableId, judgeId, targetId]
  │
  └─→ TenantModule ←─→ Module                        // 机构-模块 多对多授权

Module (模块定义) ─── 平台级，无 tenantId，超管维护
```

### 8.3 权限与多租户设计

#### 多租户隔离策略

```
┌─────────────────────────────────────────────────────┐
│  请求进入                                             │
│    ↓                                                 │
│  JWT 解析 → 提取 userId + tenantId + role             │
│    ↓                                                 │
│  TenantGuard 中间件                                   │
│    ├─ SUPER_ADMIN → 可跨租户操作（超管后台）            │
│    └─ 其他角色 → 强制注入 tenantId 到所有查询           │
│    ↓                                                 │
│  Prisma 查询层 → WHERE tenantId = ? 自动附加           │
│    ↓                                                 │
│  数据返回（已隔离）                                     │
└─────────────────────────────────────────────────────┘
```

核心规则：
- **tenantId 从 JWT 提取**，不从请求参数传入，防止伪造
- **除 `modules` 表外**，所有业务表都有 tenantId 字段
- **SUPER_ADMIN 是唯一可跨租户的角色**，用于超管后台管理所有机构
- **复合唯一约束**：如 `@@unique([tenantId, username])` 确保同机构内用户名唯一

#### 角色权限矩阵

| 操作 | SUPER_ADMIN | TENANT_ADMIN | TEACHER | STUDENT |
|------|:-----------:|:------------:|:-------:|:-------:|
| 管理所有机构 | ✅ | ❌ | ❌ | ❌ |
| 模块授权管理 | ✅ | ❌ | ❌ | ❌ |
| 管理本机构用户/组织 | ✅ | ✅ | ❌ | ❌ |
| 题库管理（CRUD） | ✅ | ✅ | ✅ | ❌ |
| 组卷/发布考试 | ✅ | ✅ | ✅ | ❌ |
| 评分打分 | ✅ | ✅ | ✅ | ❌ |
| 参加考试/答题 | ❌ | ❌ | ❌ | ✅ |
| 查看自己成绩 | ❌ | ❌ | ❌ | ✅ |
| 查看班级/全校成绩 | ✅ | ✅ | ✅ | ❌ |

#### 模块权限控制

```
用户访问模块界面
  ↓
路由守卫检查 → tenant_modules 表是否有该模块授权
  ├─ 未授权 → 隐藏入口，拒绝访问
  ├─ 已过期 → 提示续费
  └─ 已授权 → 放行，加载模块路由
```

### 9.2 核心功能操作步骤

#### 操作流程 1：教师出题 → 组卷 → 发布考试（最核心流程）

```
步骤1：创建题目分类
  教师进入「题库管理」→ 左侧分类树点击「新增分类」→ 填写分类名称 → 保存
  （支持多级分类，可设置 moduleType 标记模块专属分类）

步骤2：录入题目
  在分类下点击「新增题目」→ 选择题型（单选/多选/判断/填空）
  → 填写题目内容 → 添加选项并勾选正确答案 → 设置难度和默认分值 → 保存
  （也可通过「批量导入」上传 Excel 模板一次性导入）

步骤3：创建试卷
  进入「试卷管理」→ 点击「新建试卷」→ 填写标题和考试时长
  → 从题库搜索添加题目（可按分类/题型/难度筛选）
  → 调整题目顺序和单题分值 → 系统自动计算总分 → 保存

步骤4：发布考试
  进入「考试管理」→ 点击「新建考试」→ 关联已创建的试卷
  → 设置考试时间（开始/结束）→ 指定考生（按用户/组织/班级选择）
  → 设置规则（允许迟交、最大切屏次数）→ 发布

步骤5：学生参加考试
  学生登录 → 「我的考试」看到待考列表 → 点击进入考试
  → 答题界面（左侧题目导航 + 顶部倒计时 + 右侧答题区）
  → 逐题作答（每题实时保存）→ 交卷（手动或超时自动）

步骤6：查看成绩
  系统自动阅卷 → 教师在「成绩查询」查看班级成绩/排名/正确率
  学生在「成绩查询」查看个人成绩/答题明细/错题回顾
```

```
步骤3：查看评分结果
  返回评分表列表 → 点击「查看记录」→ 查看所有打分记录
```

### 10.1 模块 0：硬件通信标准模块（一期已完成）

**模块编码**：`HARDWARE_BRIDGE`
**类型**：基础设施层（不面向用户，为其他硬件模块提供底层能力）
**分期**：一期 Sprint 9 ✅

#### 定位
统一抽象层，上层模块无需关心底层协议。提供 Web Serial + Web Bluetooth 双通道设备连接、数据帧解析/编码、设备管理器（连接池/重连/事件分发）。

#### 功能子模块

| 子模块 | 文件 | 职责 |
|--------|------|------|
| 设备抽象接口 | `IHardwareDevice.ts` / `IDataFrame.ts` | 定义统一的设备连接/断开/收发接口 |
| 设备管理器 | `HardwareManager.ts` | 连接池管理、自动重连、事件分发 |
| 蓝牙适配器 | `BluetoothDevice.ts` / `BluetoothScanner.ts` | Web Bluetooth API 封装 |
| 串口适配器 | `SerialDevice.ts` / `SerialPortSelector.ts` | Web Serial API 封装 |
| 协议解析 | `FrameParser.ts` / `FrameEncoder.ts` | 数据帧解析/编码（支持自定义协议） |
| 调试面板 | `HardwareDebugPanel.vue` | 连接状态/收发日志/手动发送（仅超管可见） |
| 设备模拟器 | `DeviceSimulator.ts` | 无硬件时的模拟数据源，用于开发调试 |

#### 菜单树（仅超管可见）

```
超管后台 → 硬件调试 /hardware
  ├── 设备扫描（蓝牙/串口）
  ├── 连接管理（已连接设备列表）
  ├── 收发日志（实时数据帧展示）
  ├── 手动发送（调试用）
  └── 模拟器开关（开发环境）
```

#### 与母本对接点
- 不直接与母本业务表交互
- 作为 npm workspace 包（`hardware-bridge`）被前端引用
- 上层模块通过 `HardwareManager` 统一调用，无需关心蓝牙/串口差异

#### 技术要点
- Web Serial / Web Bluetooth 仅 Chromium 内核支持，必须 HTTPS
- 设备模拟器在 `NODE_ENV=development` 时自动启用，生产环境禁用
- 协议解析器支持插件式扩展，每个硬件模块注册自己的帧格式

#### 核心操作流程

```
流程1：经络描绘练习
  进入「实操练习」→ 经络描绘 → 选择经络（如手太阴肺经）
  → 在 3D 人体表面绘制经络线条（可旋转模型/回撤线条）
  → 提交 → 系统自动评分（对比标准循行路径）
  → 显示标准路径 + 得分 → 同步母本

流程2：针灸处方考核
  教师发起处方考核 → 学生进入考核界面
  → 查看病例信息 → 选择主穴/配穴 → 在 3D 模型上虚拟针刺
  → 选择针具/角度/深度/手法 → 提交
  → 系统评分（选穴合理性/针刺顺序/定位准确性/手法规范性）
  → 成绩同步母本

流程3：硬件联动教学
  连接穴位触控笔 → 用触控笔点触实体模型穴位
  → 模型穴位灯光亮起 → 软件 3D 模型同步跳转到对应穴位
  → 语音播报穴位名称 → 显示穴位详情
```

#### 与母本对接点

| 对接项 | 方式 |
|--------|------|
| 硬件 | `hardware-bridge`（触控笔通过蓝牙连接） |
| 3D 模型 | 与针刺/推拿模块共用人体模型资源 |
| 考核 | 母本 `exams` + `scores`，`moduleCode = 'MERIDIAN_ANATOMY'` |
| 练习数据 | 新增模块专属表（描绘记录/取穴记录/处方记录），评分同步母本 |

#### 技术要点
- 3D 人体模型需要支持多层透明度（皮肤/肌肉/骨骼 0-100%），Three.js 材质控制
- 经络描绘需要在 3D 表面绘制线条（Three.js Raycaster + 曲线绘制）
- 触控笔蓝牙通信需要自定义 GATT 服务/特征值解析
- 穴位灯光控制需要双向通信（软件→硬件控制灯光，硬件→软件报告触控事件）
- 模型资源可与针刺模块（10.6）和推拿模块（10.8）共用，统一存储在 OSS

| 功能 | 状态 | 说明 |
|------|------|------|
| 断网续答 | ✅ | 网络断开时答案保存在本地，网络恢复后自动同步 |

**改动文件**：
- 前端：`ExamRoomView.vue`（网络状态监听、localStorage 缓存、待同步队列、自动同步）

**实现逻辑**：
```
1. 答题时先保存到 localStorage
2. 在线时尝试同步到后端，失败则加入待同步队列
3. 离线时答案保存在本地，显示离线提示条
4. 网络恢复时自动同步待同步答案（每 5 秒尝试一次）
5. 交卷前强制同步所有待答案，同步后清除本地缓存
```

**用户提示**：
- 离线时：顶部显示黄色警告条「网络已断开，答案将保存在本地」
- 恢复时：显示绿色提示「网络已恢复，正在同步答案…」
- 同步中：显示蓝色提示条「正在同步答案…还有 N 道题待同步」

**缓存清理**：
- 交卷成功后自动清除 localStorage 缓存
- 防止下次进入考试时加载旧答案
