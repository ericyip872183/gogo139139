## 一、新开一个项目

### 1.1 流程概览

```
策划书 → 全局地图确认 → 技术选型 → 数据库设计
→ 菜单与操作逻辑 → Sprint 计划 → 生成所有文档文件
```

> 核心原则：没有全局地图之前，不做任何细化。每一步都要等你确认后再进行下一步。

---

### 第零步：生成全局地图（必须最先做）

把策划书完整给 AI，让它先输出整体鸟瞰图，你确认方向对了，后面所有步骤才基于这张图展开。

**你说：**
```
我有一份产品策划书，请先不要做任何细化，
只帮我输出一张「全局地图」，让我确认整体方向是对的。

全局地图包含：
1. 产品一句话定位
2. 所有用户角色（是谁在用，各自能做什么）
3. 所有功能模块列表（只列名字，不展开）
4. 模块之间的关系（哪些模块互相依赖）
5. 功能分期建议（一期/二期/三期，说明理由）
6. 最大的技术风险点（提前预警）

策划书内容：
【粘贴你的策划书】

输出完后等我确认，不要自动往下做。
```

**AI 应该输出：**
```
产品定位：一句话说清楚这个产品是什么

用户角色：
- 超级管理员：管理所有机构和用户
- 机构管理员：管理本机构的数据
- 学生：参加考试、查看成绩

功能模块：
- 用户管理、组织架构、题库管理、考试管理、成绩查询...

分期建议：
- 一期：登录、用户管理、题库、考试、成绩（核心链路）
- 二期：评分表、硬件对接、PBL 讨论
- 三期：数据统计、远程升级

风险点：
- 硬件对接需要 HTTPS，本地开发需要特殊处理
- 多租户隔离复杂度高，需要提前设计好

---
以上是全局地图，请确认方向是否正确？
```

**你确认后说：**
```
全局地图确认，继续第一步技术选型
```

---

### 第一步：技术选型

**你说：**
```
全局地图已确认，现在帮我定技术选型。
我的情况：【说明团队规模，比如「只有我一个人开发」或「3 人团队，熟悉 Vue」】

请输出：
1. 前端框架（说明为什么选这个）
2. 后端框架（说明为什么）
3. 数据库（说明为什么）
4. 是否需要缓存，用什么
5. 文件存储方案
6. 本地开发环境（Docker 怎么配）
7. 生产部署方案（PM2+Nginx 还是全容器化）
8. 有哪些技术难点需要提前考虑

每个选择说明理由，并告诉我有没有更简单的替代方案。
```

---

### 第二步：数据库设计

**你说：**
```
技术选型已确认，帮我设计数据库。

请输出：
1. 所有核心表（每张表一句话说清用途）
2. 表与表之间的关系
3. 权限/多租户怎么设计
4. 用 Prisma schema 格式输出完整结构

原则：考虑未来扩展，但不要过度设计。
```

---

### 第三步：菜单结构与操作逻辑

**你说：**
```
数据库已确认，帮我设计菜单结构和操作逻辑。

请输出：
1. 每个角色登录后的菜单树（树形结构）
2. 核心功能的操作步骤（用户怎么完成一个主要任务）
3. 页面跳转关系（从哪里进，到哪里去）
4. 哪些页面/按钮需要权限控制

用树形结构 + 文字流程描述，不需要画图。
```

---

### 第四步：Sprint 开发计划

**你说：**
```
菜单和逻辑已确认，帮我制定 Sprint 开发计划。

要求：
1. 按 Sprint 划分，每个 Sprint 2 周
2. 每个 Sprint 的具体任务（后端和前端分开列）
3. 每个 Sprint 的验收标准（怎么算做完）
4. 模块之间的依赖顺序（哪个必须先做）
5. 整体预计周期

原则：先做核心链路，再做周边功能。
```

---

### 第五步：生成所有文档和配置文件

**你说：**
```
以上全部确认，现在按顺序生成所有文件，每个文件生成后等我确认再继续：

1. CLAUDE.md（项目规范）
2. docs/PROGRESS.md（功能进度表，含 Sprint 计划）
3. docs/INDEX.md（文档索引）
4. docs/ARCHITECTURE.md（架构设计）
5. docs/DATABASE.md（数据库设计）
6. docs/API.md（接口规范）
7. docs/DEPLOYMENT.md（部署说明）
8. docker-compose.yml（本地开发，只跑 MySQL+Redis）
9. docker-compose.prod.yml（生产环境，全套服务）
10. .env.example（环境变量模板）
11. deploy/sync-to-server.sh（一键同步脚本）
```

---

### 1.2 所有项目统一的文档结构

```
项目根目录/
├── CLAUDE.md                   # 项目规范（AI 每次必读）
├── docs/
│   ├── PROGRESS.md             # 功能进度表（必须有）
│   ├── INDEX.md                # 文档索引（必须有）
│   ├── ARCHITECTURE.md         # 架构设计（必须有）
│   ├── DATABASE.md             # 数据库设计（必须有）
│   ├── API.md                  # 接口规范（必须有）
│   ├── DEPLOYMENT.md           # 部署说明（必须有）
│   └── MODULES_*.md            # 模块设计（按需）
├── docker-compose.yml          # 本地开发（必须有）
├── docker-compose.prod.yml     # 生产环境（必须有）
├── .env.example                # 环境变量模板（必须有）
└── deploy/
    ├── nginx.conf
    ├── sync-to-server.sh       # 一键同步脚本
    └── deploy.sh               # 一键部署脚本
```

### 1.3 端口分配规范

> **使用说明**：本节内容是**给新开项目时 AI 看的指令**。
> 新开项目时，把下方「新项目端口分配指令」整段复制给 AI，它会自动查端口分配记录，选好端口后更新记录。

---

#### 全局端口分配记录文件

**位置**：`C:\项目\端口分配记录.md`

**作用**：记录所有项目的端口使用情况，新项目必须避开已占用端口。

**每次新开项目时**：
1. AI 先读取此文件，查看哪些端口已被占用
2. 按规则选择未被占用的端口
3. 选好后立即更新此文件，写入新项目信息

**你**（用户）：每次项目部署到服务器后，更新生产环境的端口信息。

---

#### 新项目端口分配指令（新开项目时复制整段给 AI）

```
请帮我分配新项目的端口，按以下步骤：

1. 先读取 C:\项目\端口分配记录.md，查看哪些端口已被占用

2. 按以下规则选择端口：
   - 本地前端：从 5173 开始，找第一个未被占用的
   - 本地后端：从 3000 开始，找第一个未被占用的
   - 生产后端：从 3002 开始，找第一个未被占用的
   - 避开系统保留端口：80/443/22/3306/6379

3. 选好后，更新 C:\项目\端口分配记录.md，写入新项目信息

4. 把选好的端口写入：
   - .env.example
   - docker-compose.yml
   - 不要硬编码在代码里

当前项目信息：
- 项目名称：【填写】
- 团队情况：【如「只有我一个人开发」】
```

---

#### 系统保留端口（所有项目不得使用）

| 端口 | 被谁占用 | 说明 |
|------|---------|------|
| 80 | HTTP | 标准网页访问端口，生产 Nginx 用 |
| 443 | HTTPS | 标准加密网页端口，生产 Nginx 用 |
| 22 | SSH | 服务器远程登录，绝对不能占用 |
| 3306 | MySQL | 数据库默认端口 |
| 6379 | Redis | 缓存默认端口 |
| 1024 以下 | 系统保留 | 普通程序不得使用 |

---

#### 端口分配示例（panlei 项目）

| 服务 | 本地端口 | 生产端口 |
|------|---------|---------|
| 前端 | 5173 | 80 / 443 |
| 后端 | 3000 | 3002 |
| MySQL | 3306 | 3306（不对外） |
| Redis | 6379 | 6379（不对外） |

## 二、回到已有项目继续开发

### 2.1 离开项目前要做的事（下次才能顺利接上）

每次离开项目前，做这两件事：

**第一件：确保代码已提交**
```
帮我提交
```
未提交的代码下次 AI 看不到，Git 记录是 AI 了解进度的唯一依据。

**第二件：在 Git commit message 里留下"下一步"**

如果当前功能做到一半，提交时说清楚：
```
帮我提交，备注下一步要做「xxx」
```

AI 会在 commit message 里加一行 `TODO: 下一步做 xxx`，下次回来一看 Git 记录就知道从哪里继续。

**真实示例：**
```
你说：帮我提交，备注下一步要做「评分表的离线同步功能」

AI 提交：
feat: 完成评分表基础 CRUD (v1.3.0)

TODO: 下一步做离线评分同步（Service Worker + 批量上传接口）
```

---

### 2.2 回到项目时的开场白

只需要说一句话，AI 会自动读文档、看 Git、告诉你当前状态：

```
继续这个项目，按规范来
```

AI 收到后必须做：
1. 读 `CLAUDE.md` — 了解项目规范和技术选型
2. 读 `docs/PROGRESS.md` — 了解功能完成状态
3. 看 Git 最新提交 — 了解最近改动和下一步 TODO
4. 回复你：当前状态摘要 + 建议下一步

---

### 2.3 真实对话示例

**你说：**
```
继续这个项目，按规范来
```

**AI 回复：**
```
已读取项目文档，当前状态：

已完成：Sprint 1-8（登录/用户/题库/考试/成绩/评分表/超管后台）
最近提交：v1.3.0 - 完成评分表基础 CRUD
TODO（上次留的）：下一步做离线评分同步（Service Worker + 批量上传接口）

建议下一步：开始离线评分同步，先做 Service Worker 缓存层。
要继续吗？
```

## 三、完成功能或修好 BUG 后提交代码

### 3.1 提交前是否需要构建？

| 情况 | 要不要构建 | 原因 |
|------|-----------|------|
| 只改了文档/配置 | 不需要 | 没有代码变动 |
| 功能开发完成，准备提交 | **建议先构建验证** | 确认没有编译错误再提交，避免提交坏代码 |
| 要部署到服务器 | 不需要手动构建 | 部署脚本里会自动构建 |

**功能完成后的标准流程：**
```
帮我先构建验证一下，没问题再提交
```
AI 会先跑 `vite build`，构建通过后再提交。如果构建失败，先修错误再提交。

---

### 3.2 提交代码的话术

**正常情况（AI 全程参与开发）：**
```
帮我提交
```
AI 已经知道改了什么，自己查 git status、写 commit message、定版本号，给你确认后提交。

**特殊情况（你自己手动改了代码，AI 不知道）：**
```
我手动改了 xxx，帮我提交，版本 v1.2.9
```
只有这种情况才需要你补充说明。

---

### 3.3 真实对话示例（2 轮完成）

**第 1 轮 — 你说：**
```
帮我提交
```

**AI 自己查改动，回复：**
```
改动文件：
- MainLayout.vue（overflow: hidden）
- AdminView.vue（tabs 布局重构）

commit: fix: 修复超管后台双滚动条问题 (v1.2.9)

确认提交？
```

**第 2 轮 — 你说：**
```
确认
```

**AI 执行，一句话报结果：**
```
✅ 已提交 v1.2.9
```

---

### 3.4 Commit Message 格式规范

```
<类型>: <描述> (<版本号>)

详细说明（可选）：
- 改动了什么
- 解决了什么问题
- 怎么用
```

**类型对照表：**

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 新增超管后台机构管理 (v1.2.8)` |
| `fix` | 修复 BUG | `fix: 修复双滚动条问题 (v1.2.9)` |
| `chore` | 配置/依赖/清理 | `chore: 移除前端类型检查 (v1.2.9)` |
| `docs` | 文档更新 | `docs: 更新开发入门教程 (v1.2.7)` |
| `cleanup` | 清理冗余文件 | `cleanup: 删除重复文档和 PDF (v1.2.9)` |
| `refactor` | 重构代码 | `refactor: 重构用户权限模块 (v1.3.0)` |

---

### 3.5 版本号规范

```
v<大版本>.<小版本>.<补丁>
```

| 位置 | 什么时候升 | 示例 |
|------|-----------|------|
| 大版本 | 架构级改动，不向下兼容 | v1.0.0 → v2.0.0 |
| 小版本 | 新增功能 | v1.2.0 → v1.3.0 |
| 补丁 | 修复 BUG、优化、清理 | v1.2.8 → v1.2.9 |

## 四、部署到服务器

### 4.1 部署话术

**标准版：**
```
帮我部署到服务器
```

**详细版：**
```
帮我部署到服务器，按以下步骤：
1. 先列出可以清理的旧文件，等我确认后再删
2. 提交最新代码
3. 运行部署脚本
4. 告诉我部署结果和访问地址
```

---

### 4.2 真实对话示例

**你说：**
```
帮我部署到服务器，先列旧文件让我确认
```

**AI 回复：**
```
发现以下可清理的文件：

| 文件 | 原因 |
|------|------|
| 医学_功能清单 V3.pdf | PDF 不应放代码仓库，内容已整合进 CLAUDE.md |
| docs/AI 协作规范.md | 已整合进 CLAUDE.md |
| docs/沟通模板.md | 已整合进 CLAUDE.md |

确认删除这些文件吗？
```

**你说：**
```
确认
```

**AI 执行并回复：**
```
✅ 已删除 3 个文件
✅ 已提交 cleanup: 清理冗余文档 (v1.2.9)
🔨 正在构建前端...
📤 正在上传到服务器...
🔄 正在重启后端服务...

✅ 部署完成
正式地址：https://pl.7025700.vip
测试地址：http://120.77.144.5:8090
```

---

### 4.3 部署前检查清单

每次部署前 AI 必须确认：

- [ ] 旧文件已列出并经用户确认
- [ ] 代码已提交并打标签
- [ ] 本地构建成功
- [ ] 服务器文件上传成功
- [ ] 服务重启成功
- [ ] 访问地址可以打开

---

### 4.4 文件清理规范

**可以清理的文件：**

| 类型 | 说明 |
|------|------|
| 重复文档 | 内容已整合进主文档的子文档 |
| PDF 文件 | 不应放代码仓库 |
| 敏感文件 | .env、*密钥*、*password*.txt |
| 临时文件 | *.tmp、*.bak、dump.sql |
| 大文件 | >10MB 的非代码文件 |

**清理原则：必须先列出来让用户确认，不能直接删。**

## 五、纠正 AI 的错误

### 5.1 为什么要纠正 AI

AI 不是万能的，它可能：
- 理解错了你的需求，做了不对的东西
- 写了有 BUG 的代码，你发现后自己改了
- 给出了错误的技术建议，你查资料后发现不对
- 记忆了错误的信息，下次还会犯同样的错

**如果你改了但不告诉 AI，下次它还会犯同样的错。**

---

### 5.2 三种纠正场景

---

#### 场景一：AI 写的代码有问题，你自己改好了

**你说：**
```
你之前写的「xxx」有问题，我已经改好了。
错误原因是：【说明什么地方错了】
正确做法是：【说明你怎么改的】
请记住这个正确做法，以后遇到类似情况不要再犯同样的错。
```

**真实示例：**
```
你之前写的 docker exec 命令有问题，在 Git Bash 里 /F 参数会被识别成路径。
错误写法：taskkill /F /PID 26836
正确做法：用 cmd /c "taskkill /F /PID 26836" 包一层 cmd 来执行。
请记住这个，以后在 Windows 环境下执行 Windows 命令要用 cmd /c 包裹。
```

**AI 应该回复：**
```
已记住。在 Windows Git Bash 环境下执行 taskkill 等 Windows 原生命令，
需要用 cmd /c "命令" 包裹，否则参数会被识别为路径。
感谢纠正。
```

---

#### 场景二：AI 给出了错误的技术建议，你查资料后发现不对

**你说：**
```
你之前说「xxx」是错的。
正确的是：【说明正确的内容】
原因是：【说明为什么】
以后遇到这类问题请用正确的方式回答。
```

**真实示例：**
```
你之前说 pnpm prisma migrate dev 可以直接用，但实际上
普通用户没有创建 shadow database 的权限，会报 P3014 错误。
正确做法是用 pnpm prisma db push，它不需要 shadow database。
以后遇到权限受限的数据库环境，优先推荐 db push 而不是 migrate dev。
```

**AI 应该回复：**
```
已更正。在数据库用户权限受限（无法创建数据库）的环境下，
应使用 prisma db push 而不是 prisma migrate dev。
prisma db push 直接推送 schema 变更，不需要 shadow database。
感谢纠正。
```

---

#### 场景三：AI 理解错了需求，做了不对的东西

**你说：**
```
你理解错了我的需求。
我要的是：【说明你真正想要的】
你做的是：【说明 AI 做了什么】
区别在于：【说明两者的差异】
请重新做。
```

**真实示例：**
```
你理解错了。
我要的是：只修复超管后台的双滚动条，其他页面不要动。
你做的是：修改了 MainLayout 的全局样式，影响了所有页面。
区别在于：应该只改 AdminView 的局部样式，不要动全局布局。
请撤销对 MainLayout 的修改，只在 AdminView 里解决这个问题。
```

---

### 5.3 纠正后要求 AI 更新记忆

如果这个错误是 AI 的**认知问题**（不是一次性的代码错误），纠正后要求 AI 把正确信息写入**记忆文件**，确保下次会话也不会犯同样的错。

**什么是记忆文件？**

| 问题 | 答案 |
|------|------|
| 位置 | `C:\Users\<你的用户名>\.claude\projects\c-----panlei\memory\MEMORY.md` |
| 大小限制 | **前 200 行有效**，超过的行会被截断不读 |
| 作用 | Claude 每次新会话自动读取，记住你的偏好和项目特殊规则 |
| 内容 | 只记录高频易错点、用户偏好、项目特殊规则 |

**因为有限制，所以必须精简。判断标准：**

| 值得记（优先写） | 不值得记（浪费行数） |
|-----------------|---------------------|
| 用户明确说的偏好（"不要用 emoji"） | 具体某次代码怎么写 |
| 项目特殊规则（"所有接口必须带 tenantId"） | 临时的调试信息 |
| 反复犯的错误纠正（"Windows 下用 cmd /c 包裹命令"） | 通用技术知识（AI 本来就知道） |
| 团队技术选型（"本项目用 Vue3+TS"） | 可能会变的信息 |

**你说：**
```
请把这个正确做法写入你的记忆文件，确保下次会话也记得。
```

**真实示例：**
```
你之前说的 prisma migrate dev 在权限受限环境下会报错，正确是用 db push。
请把这个写入你的记忆文件，以后不要再犯。
```

**AI 应该回复并执行：**
```
已写入记忆文件。以后在这个项目中，遇到 Prisma 数据库同步：
- 权限受限环境（用户无创建数据库权限）→ 用 prisma db push
- 完整权限环境 → 可以用 prisma migrate dev
```

---

### 5.4 纠正的原则

| 原则 | 说明 |
|------|------|
| 说清楚错在哪 | 不要只说「不对」，要说具体哪里错了 |
| 说清楚正确做法 | 告诉 AI 正确答案，不要让它自己猜 |
| 说清楚原因 | 让 AI 理解为什么，而不只是记住结论 |
| 要求记住 | 明确说「以后遇到类似情况不要再犯」 |
| 重要的写入记忆 | 如果是反复会犯的错，要求写入记忆文件 |

---

### 5.5 不需要纠正的情况

- AI 只是表达方式不同，但结果是对的 → 不需要纠正
- 一次性的小失误，不影响后续 → 不需要纠正
- 你自己改了一行代码，AI 不需要知道 → 不需要纠正

**只有当错误会影响后续开发，或者 AI 会反复犯同样的错时，才需要正式纠正。**

## 六、规范参考

### 6.1 快捷话术速查

| 场景 | 说这句话 |
|------|---------|
| 回到项目 | `继续这个项目，按规范来` |
| 提交新功能 | `帮我提交，新功能「xxx」，版本升到 v1.x.x` |
| 提交 BUG 修复 | `帮我提交，修好了「xxx」问题，版本升到 v1.x.x` |
| 部署 | `帮我部署到服务器` |
| 新开项目 | `我要开新项目，先帮我生成全局地图，策划书如下：` |
| 纠正 AI | `你之前说的「xxx」是错的，正确做法是「xxx」，请记住` |

---

### 6.2 AI 每次必须做的事

**回到项目时：**
1. 读 `CLAUDE.md` — 了解项目规范
2. 读 `docs/PROGRESS.md` — 了解功能完成状态
3. 看 Git 最新提交 — 了解最近改动
4. 告诉你当前状态，再问你要做什么

**提交代码时：**
1. 先列出改动的文件
2. 写好 commit message 给你确认
3. 确认后才执行提交和打标签
4. 如果是新功能，同步更新 `docs/PROGRESS.md`

**部署时：**
1. 先列出可清理的旧文件给你确认
2. 确认后删除并提交
3. 执行部署脚本
4. 报告结果和访问地址

**被纠正时：**
1. 承认错误，说清楚哪里错了
2. 确认正确做法
3. 如果要求写入记忆，立即更新记忆文件
4. 不要辩解，不要重复犯同样的错
